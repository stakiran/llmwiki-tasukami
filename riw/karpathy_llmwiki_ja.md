# LLMウィキ

LLMを使って個人のナレッジベースを構築するためのパターン。

これはアイデアを記したファイルであり、自分のLLMエージェント（OpenAI Codex、Claude Code、OpenCode / Pi など）にコピペして使うことを想定している。目的は高レベルの構想を伝えることであり、具体的な部分はあなたのエージェントがあなたと協力しながら作り上げていく。

## 中核となるアイデア

ほとんどの人にとってのLLMと文書の付き合い方はRAGのようなものだ。ファイル群をアップロードし、クエリ時にLLMが関連するチャンクを取り出し、回答を生成する。これはこれで機能するが、LLMは質問のたびにゼロから知識を再発見している。蓄積がない。5つの文書を統合する必要がある微妙な質問をすれば、LLMは毎回該当する断片を探し出してつなぎ合わせなければならない。何も積み上がっていかない。NotebookLM、ChatGPTのファイルアップロード、そしてほとんどのRAGシステムはこの方式だ。

ここで提案するアイデアは別物だ。クエリ時に生のドキュメントから取り出すだけでなく、LLMが**永続的なウィキを段階的に構築・維持していく** — 構造化され、相互にリンクされたマークダウンファイル群が、あなたと生のソースとの間に介在する。新しいソースを追加すると、LLMはそれを後で取り出すためのインデックスに加えるだけではない。読み込み、要点を抽出し、既存のウィキに統合する — エンティティページを更新し、トピックの要約を改訂し、新しいデータが古い主張と矛盾する箇所をメモし、進化していく統合像を補強したり挑戦したりする。知識は一度コンパイルされ、その後*最新の状態に保たれ続ける* — クエリのたびに再導出されるのではない。

ここが決定的な違いだ。**ウィキは永続的で、複利的に成長していく成果物である。**相互参照はすでに張られている。矛盾はすでに指摘されている。統合は、あなたが読んだすべてをすでに反映している。ソースを追加するたび、質問するたびに、ウィキはどんどん豊かになっていく。

ウィキを自分で書くことは（ほとんど）ない — LLMがすべてを書き、保守する。あなたの担当は、ソースの調達、探索、そして適切な質問をすることだ。LLMは要約、相互参照、ファイリング、簿記といった、ナレッジベースを長期的に有用たらしめる地味な作業すべてを引き受ける。実際の使い方としては、私はLLMエージェントを片側に開き、Obsidianを反対側に開いている。LLMが会話に基づいて編集を行い、私はその結果をリアルタイムで眺める — リンクをたどり、グラフビューを確認し、更新されたページを読む。ObsidianはIDEで、LLMはプログラマー、ウィキはコードベースだ。

これはさまざまな文脈に適用できる。いくつか例を挙げる。

- **個人**：自分自身の目標、健康、心理、自己改善を追跡する — 日記のエントリ、記事、ポッドキャストのメモを整理し、自分自身の構造化された姿を時間をかけて構築していく。
- **リサーチ**：あるテーマを数週間〜数ヶ月かけて深掘りする — 論文、記事、レポートを読み込み、進化していくテーゼを伴った包括的なウィキを段階的に構築していく。
- **読書**：章を読み進めるごとにファイリングし、登場人物・テーマ・プロット・それらの関係についてのページを構築していく。読み終わるころには豊かなコンパニオンウィキが手元にある。[Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) のようなファンウィキを思い浮かべてほしい — ボランティアコミュニティが何年もかけて、登場人物、場所、出来事、言語をカバーする数千ものリンクされたページを構築している。LLMがすべての相互参照と保守を行えば、あなたが読書をする中で個人的にそういうものを作り上げられる。
- **ビジネス／チーム**：LLMによって維持される社内ウィキ。Slackのスレッド、会議の文字起こし、プロジェクト文書、顧客との通話などが供給される。場合によっては人間がループに入って更新内容をレビューする。チームの誰もやりたがらない保守作業をLLMが行うことで、ウィキは最新の状態に保たれる。
- **競合分析、デューデリジェンス、旅行計画、講義ノート、趣味の深掘り** — 知識を時間をかけて蓄積し、散らばっているのではなく整理しておきたいもの全般。

## アーキテクチャ

3つの層がある。

**生のソース** — あなたが厳選したソース文書のコレクション。記事、論文、画像、データファイル。これらは不変だ — LLMは読み取るだけで、決して変更しない。これがあなたの真実の源泉となる。

**ウィキ** — LLMが生成したマークダウンファイルが置かれるディレクトリ。要約、エンティティページ、概念ページ、比較、概観、統合。LLMがこの層を完全に所有する。ページを作成し、新しいソースが届けば更新し、相互参照を維持し、すべてに整合性を持たせる。あなたは読み、LLMが書く。

**スキーマ** — ウィキがどう構成されているか、どんな規約があるか、ソースの取り込み・質問への回答・ウィキの保守の際にどんなワークフローに従うべきかをLLMに伝える文書（Claude CodeならCLAUDE.md、CodexならAGENTS.md など）。これが鍵となる設定ファイルだ — これがあるおかげで、LLMは汎用的なチャットボットではなく、規律あるウィキ保守者として機能する。あなたとLLMは時間をかけて、ドメインに合わせて何が機能するかを共に進化させていく。

## 操作

**取り込み（Ingest）。** 新しいソースを生のコレクションに放り込み、LLMに処理を指示する。たとえば、こんな流れになる：LLMがソースを読み、あなたと要点について議論し、ウィキに要約ページを書き、インデックスを更新し、ウィキ全体の関連するエンティティページや概念ページを更新し、ログに項目を追加する。ひとつのソースで10〜15のウィキページに手が入ることもある。個人的には、私はソースを一つずつ取り込み、関与し続けるやり方を好む — 要約を読み、更新を確認し、何を強調すべきかLLMにガイドする。だが、監督を緩めて多数のソースを一気に取り込むこともできる。自分のスタイルに合うワークフローを開発し、それを将来のセッション用にスキーマに記述しておくのはあなた次第だ。

**クエリ（Query）。** ウィキに対して質問を投げる。LLMは関連するページを検索し、読み、引用付きの回答を統合する。回答は質問に応じて様々な形を取りうる — マークダウンページ、比較表、スライドデッキ（Marp）、チャート（matplotlib）、キャンバス。重要な洞察：**良い回答は新しいページとしてウィキに還元できる。**あなたが依頼した比較、分析、発見した関係性 — これらは価値があるのであり、チャット履歴に消えていくべきではない。こうすれば、取り込まれたソースと同じように、あなたの探索もナレッジベースに複利的に蓄積されていく。

**リント（Lint）。** 定期的にLLMにウィキのヘルスチェックを依頼する。チェック項目：ページ間の矛盾、新しいソースによって取って代わられた古い主張、被リンクのない孤児ページ、言及されているのに専用ページがない重要な概念、欠けている相互参照、ウェブ検索で埋められそうなデータのギャップ。LLMは新たに調査すべき問いや探すべきソースを提案するのが得意だ。これによってウィキは成長しながらも健全に保たれる。

## インデックス化とロギング

ウィキが大きくなるにつれて、LLM（とあなた）がナビゲートするのを助ける2つの特別なファイルがある。それぞれ目的が異なる。

**index.md** はコンテンツ志向だ。ウィキ内のすべてのカタログ — 各ページがリンク、一行の要約、そして必要に応じて日付やソース数といったメタデータとともに列挙される。カテゴリー別（エンティティ、概念、ソースなど）に整理される。LLMは取り込みのたびにこれを更新する。クエリに答える際、LLMはまずインデックスを読んで関連ページを探し、それから掘り下げる。これは中規模（〜100ソース、〜数百ページ）では驚くほどうまく機能し、埋め込みベースのRAGインフラを必要としない。

**log.md** は時系列だ。何がいつ起きたかを記録する追記専用のファイル — 取り込み、クエリ、リントパス。便利なコツ：各エントリを一貫したプレフィックス（例：`## [2026-04-02] ingest | Article Title`）で始めると、ログは単純なunixツールでパース可能になる — `grep "^## \[" log.md | tail -5` で直近5件のエントリが得られる。ログはウィキの進化のタイムラインを与え、最近何が行われたかをLLMが理解する助けとなる。

## オプション：CLIツール

そのうち、LLMがウィキ上でより効率的に作業できるよう、小さなツールを作りたくなるかもしれない。最も明らかなのはウィキページに対する検索エンジンだ — 小規模ならインデックスファイルで十分だが、ウィキが成長するにつれてちゃんとした検索が欲しくなる。[qmd](https://github.com/tobi/qmd) は良い選択肢だ：マークダウンファイル向けのローカル検索エンジンで、BM25とベクトル検索のハイブリッドにLLMによるリランキングが加わり、すべてオンデバイスで動く。CLI（LLMがshellから呼べる）とMCPサーバ（LLMがネイティブツールとして使える）の両方を備えている。あるいは、もっとシンプルなものを自作することもできる — 必要に応じてLLMに手伝わせて、素朴な検索スクリプトを vibe-code で書けばいい。

## 小ネタとコツ

- **Obsidian Web Clipper** はWeb記事をマークダウンに変換するブラウザ拡張だ。生のコレクションにソースを素早く取り込むのに非常に便利。
- **画像をローカルにダウンロードする。** Obsidian の Settings → Files and links で、"Attachment folder path" を固定のディレクトリ（例：`raw/assets/`）に設定する。次に Settings → Hotkeys で "Download" を検索して "Download attachments for current file" を見つけ、ホットキー（例：Ctrl+Shift+D）に割り当てる。記事をクリップした後にホットキーを押すと、すべての画像がローカルディスクにダウンロードされる。これは任意だが便利だ — LLMが、壊れる可能性のあるURLに頼らず、画像を直接見たり参照したりできるようになる。LLMはインライン画像つきのマークダウンを一発でネイティブに読めるわけではない点に注意 — 回避策は、まずテキストを読ませ、それから参照されている画像の一部または全部を別途見させて追加コンテキストを得ることだ。少し不格好だが、十分機能する。
- **Obsidianのグラフビュー** はウィキの形を見るのに最適 — 何が何につながっているか、どのページがハブで、どれが孤児か。
- **Marp** はマークダウンベースのスライドデッキ形式。Obsidianにプラグインがある。ウィキの内容から直接プレゼンを生成するのに便利。
- **Dataview** はページのフロントマターに対してクエリを走らせるObsidianプラグインだ。LLMがウィキページにYAMLフロントマター（タグ、日付、ソース数）を付加してくれれば、Dataviewが動的なテーブルやリストを生成できる。
- ウィキは結局のところマークダウンファイルのgitリポジトリに過ぎない。バージョン履歴、ブランチ、共同作業がタダで手に入る。

## なぜこれが機能するのか

ナレッジベースを保守する上で面倒なのは、読むことや考えることではなく、簿記作業だ。相互参照の更新、要約を最新に保つこと、新しいデータが古い主張と矛盾するときに記しておくこと、何十ものページにわたって整合性を保つこと。人間がウィキを放棄するのは、保守の負担が価値の増加よりも速く膨らむからだ。LLMは退屈しないし、相互参照の更新を忘れず、一度に15ファイルに手を入れられる。保守のコストがほぼゼロだから、ウィキは保守され続ける。

人間の仕事は、ソースを厳選し、分析の方向を指示し、良い質問をし、そのすべてが何を意味するのかを考えることだ。LLMの仕事は、それ以外のすべてである。

このアイデアは、精神においてはVannevar Bushの「Memex」（1945）と関連している — 個人が能動的にキュレートする知識ストアで、文書間に連想的な道筋がある、というものだ。Bushのビジョンは、Webが実際になったものよりも、こちらに近かった：プライベートで、能動的にキュレートされ、文書同士のつながりが文書そのものと同じくらい価値を持つ。彼に解けなかった部分は、誰がその保守をするか、だった。LLMがそれを担う。


## ノート

この文書は意図的に抽象的だ。アイデアを述べているのであって、特定の実装を述べているわけではない。正確なディレクトリ構造、スキーマの慣習、ページのフォーマット、ツール群 — それらはすべて、あなたのドメイン、好み、選んだLLMに依存する。上で触れたものはすべて任意でモジュール式だ — 役に立つものを採用し、そうでないものは無視すればいい。たとえば：あなたのソースがテキストだけなら、画像処理はまったく不要かもしれない。ウィキが小さければインデックスファイルだけで十分で、検索エンジンは要らないかもしれない。スライドデッキには関心がなく、マークダウンページだけ欲しいかもしれない。まったく別の出力フォーマット群を望むかもしれない。これを正しく使う方法は、自分のLLMエージェントとこの文書を共有し、自分のニーズに合わせたバージョンを一緒に立ち上げていくことだ。この文書の唯一の仕事は、パターンを伝えることだ。残りはあなたのLLMが考えてくれる。
