# Issueist

GitHub Issues など **BTS（Bug Tracking System）** を使う [戦略](strategies.md)。話題で区切る系。

## 概要

- 1 タスク 1 ページ（チケット）でタスクを管理
- 元はソフトウェア開発でバグを 1 つずつ確実に管理するもの。Redmine、Backlog などプロジェクトタスク管理寄りのツール多数
- GitHub Issues は個人タスク管理にも使いやすく、エンジニアに支持される
- [Alternate（俯瞰と注視）](alternate.md) の概念は BTS から来たと著者は考えている

> Issueist は、BTS を用いて俯瞰と注視に専念する戦略

- 1 タスク 1 ページなのでじっくり向き合える
- チャットのようにテキストや画像を残せる
- フィルタリング・ソートで数百〜数千件から絞り込める
- カンバンボード機能で視覚的にも俯瞰できる

## 応用

### どこまで丁寧に運用するか
- **丁寧運用** — プロジェクトタスク管理化する。生活全般では息切れするが、大きな仕事や用事はアリ。GitHub のリポジトリ単位で「引っ越しリポジトリ」「書籍執筆リポジトリ」など
- **雑運用** — やらなくてもいいけどやりたいこと（考え事、調べ事、新しく試したいこと）を雑多に。気まぐれに眺めて気まぐれに消化。著者はこちらが多い

折衷案:

- **バックログ的** — やりたいことを Issue として全部つっこみ、余裕がある時に消化。週一・月一の棚卸し
- **インボックス的** — オープン Issue 0 件を目指す、Dailer に近い形（**インボックスゼロ**、[Inboxer](inboxer.md) 参照）

### ツール
- 特に問題なければ **GitHub Issues** 一択（リポジトリ単位、軽快）
- すでに他 BTS を使っているならそれ
- 個人で使うには料金的に割に合わない可能性
- 日本語 UI なし、初学者は厳しい

## メリット
- BTS に触れている人には馴染み深い
- 俯瞰と注視に専念できるシンプルな使い心地

## デメリット
- 初学者のハードルが高い
- 管理をどこまで厳しくするかのトレードオフが難しい
- 細かい・多数のタスクを扱いづらい（粒度大きめ向き、小回りに不向き）
- 俯瞰を機能させるのが難しい（属性入力やボードのメンテナンス）

## 適性
- **BTS に慣れている人** — GitHub Issues を使えるかが目安。エンジニアなど
- **ナレッジワーカー** — 1 検討事項 1 ページで思考を蓄積できる。「〜について考えてみる」のような気軽なタスクが作りやすい

ナレッジワーカー特有の感覚: タスクとして通常浮かべるのは「外部から降ってきたものを処理」「未来の脅威に備える」だが、ナレッジワーカーは「降ってきてもないし備えるつもりもないが考えたい」ことが多く、話題で区切れる BTS がうってつけ。
