# ETM の 4 つの管理

[ETM](etm.md) における **タスク管理以前の管理**。探索を上手くやるためには、ロード / ターゲット / リソース / コンテキストの 4 つを管理する必要がある。

ETM は探索的アプローチであるため、「終了条件の見えない曖昧なタスク」の管理が中心となる。自己制御に力点を置く。

## 1: ロードの管理

**ロード（Load）** = 直近取り組みたいこと（重視したいこと）として抱えているもの。

性質:

| 性質 | 内容 |
| --- | --- |
| **ロードキャパシティ** | 何個抱えられるかが人によって違う。**一桁**（≦9）に収まる |
| **粗さ** | タスクよりかなり粗い。「引っ越し」「転職を検討する」レベル |
| **不確定性** | 終わりは見えない。意思決定で終わらせるしかない |

ニュアンスとしては「常に背負っている荷物」。タスクではないので毎日必ず使う必要はないが、毎日使う機会はある。

### 一桁の原則

> ロードキャパシティは一桁（9 個以内）に収まる

著者体感では 9 個でも多すぎ、5 個でちょうど。超える場合はロードの捕捉が下手（タスクをロード扱いしている等）。

### ロードキャパシティの可視化
複数人で行う場合、メンバー同士でロードを見えるようにする。ロードノート / ロードページ / ホーム掲載など。

> 重要なのはロードキャパシティの可視化

プライベートなロード（婚活など）は内容を伏せて「私用」とだけ書いてもよい。皆のロードキャパシティは **空気** であり、互いの様子の把握と自分の行動への反映に役立つ。

## 2: ターゲットの管理

トピックは何百何千とつくられるため選別が必要。**3T モデル** で扱う。

### 3T モデル

トピックを 3 状態で捉え、Topic → Target → Task と昇格させる。

| 状態 | 内容 | 当事者 | 催促 |
| --- | --- | --- | --- |
| **1 Topic** | ただのトピック | なし（誰でも好き勝手に読み書き） | 不可 |
| **2 Target** | 当事者が存在するトピック | 1 名（同意があれば複数も可） | 不可 |
| **3 Task** | 当事者ありかつ催促が許される | 当事者あり | 可 |

### 当事者
**当事者** = Target 以上のトピックをリードする者。意思決定権を持つ代わりに責任を負う。それ以外は **助言者**（[ティール組織](https://www.amazon.co.jp/dp/4862762263) の助言者モデル）で意思決定権なし。

### 催促
**催促** = 忘迷怠のフォロー（「～～の件どうなってる？」「3 日間音沙汰無いけどどした？」）。

ETM では通知を飛ばさない（メンション・会議・メッセージ送信不可）ため、催促はノートやホーム動線上で行う。

### Target は表明
「私は今からこのトピックを少し重視しますよ」と告げるようなもの。**約束ではない**。

### 2 つのキャパシティ
ETM 実践者は 2 つのキャパシティを持つ:

| | 内容 | 可視化 |
| --- | --- | --- |
| **ロードキャパシティ** | 大局的な方針のキャパシティ | 必要 |
| **当事者キャパシティ** | 抱える Target の数 | 不要 |

両方とも一桁の原則が当てはまる。

### 当事者はよく変わる
A さんが諦めたら B さん、B さんから C さんへ──と動的に交代可能。

> ダメならば外せばいいだけ

凝り出すときりがないので一段落ついたらトピック A は確定とみなし、別の派生トピック B で継続する。

### 単一当事者の原則

> 当事者は可能な限り 1 人

ETM は探索的な営みであり、探索は一人で行うもの（作家のメタファー）。複数人の当事者が望ましく見える場合は、トピック粒度が粗いことが多い。分解して各自 1 人当事者になる方が上手くいく。

複数当事者を立てたい場合: 1 人目は早い者勝ち、2 人目以降は現存当事者全員の同意が必要。

### 単一当事者を阻む構造的問題
2 つある:

| 問題 | 内容 |
| --- | --- |
| **権限委譲のしづらさ** | 階層組織。マネージャーがボトルネック。対等性が必要（[ティール組織](https://www.amazon.co.jp/dp/4862762263) が解） |
| **探索が有効でない状況** | No ABCD が当てはめられない。スケジュールタイト・予算不足など |

### ブースター
**ブースター（Booster）** = 行動時に使える各種情報。行動をブーストさせる。

> ETM で行うのは、実際の行動というよりも行動可能なところまで情報を揃える部分

トピックはブースターの集積。実際の行動は ETM の範囲外。

例: 大企業で「全員に社有 iPhone を持たせるべき」というトピック → ETM の範囲はこの結論を出すところまで。実際の調達・予算反映はせず、別途行動する。

> 行動まで済ませようとするシチュエーションは ETM には合わない

## 3: リソースの管理

やる気・体力（身体面・頭脳面）・注意資源・社交エネルギーなど私たちが動くための資源を管理する。

可視化・管理されたタスクをどうリソースに振るかが本質。

### 管理 2.0

> 結果を管理するのではなく、結果を生み出すもとである私たちの状態を計測する

| | 内容 |
| --- | --- |
| **管理 1.0** | 成果物の進捗・予実 |
| **管理 2.0** | 理想状態の因子の計測（出力ではなく入力の計測） |

元ネタ: [Engineering Effectiveness](https://ee-handbook.io/)。生活では日々体調を記録する **セルフモニタリング** がこれに近い。

ETM は管理しないので 1.0 はできない。代わりに自分の状態を計測し、傾向から働きかける。

### 理想状態
ETM での定義: **自分の生活と心身を尊重しながらも、普段のパフォーマンスを発揮できること**。

ただし探索的アプローチを取るため、**管理を肯定・促進する状態**（「週一で出社して顔合わせ」など人を巻き込むもの）は除外する。理想状態は **個人に閉じる**。

### 計測因子のポイント

| ポイント | 内容 |
| --- | --- |
| 細かすぎない | ライフログのように何でも記録しても読まない |
| 厳選して一桁以内 | 数個から始める |
| 突き詰めて選定 | 「これじゃないか」と感じる候補を遠慮なく |

#### ポジティブ vs ネガティブ
- **ポジティブ因子** — 起きたらプラスになる（朝のルーチン全消化など）
- **ネガティブ因子** — 起きたらマイナスになる（雨天など）

戦略:

| 戦略 | 内容 |
| --- | --- |
| **ミニマイザー** | ネガティブを減らす。海外旅行に行きたくないタイプ |
| **マキシマイザー** | ポジティブを増やす。海外旅行に行きたいタイプ |
| **ハイブリッド** | 両方 |

> ETM のような探索的な営みでは、ネガティブな因子の計測の方が重要

ポジティブ因子は「見つかったらラッキー」程度。

#### 主要なネガティブ因子の観点

| 因子 | 内容 |
| --- | --- |
| **拘束の数** | 会議・話し込みの数 |
| **割り込みの数** | [割り込み](warp.md) を受けた数。少ない方がよい |
| **メンタルロードの数** | [メンタルロード](mental_load.md)（頭の中で高頻度にちらつく不安・不満）の個数 |
| **暴力の有無** | 物理 / 怒気 / ハラスメント / 人格批判 |
| **体調不良の有無** | 低気圧、筋肉痛、花粉、生理など |
| **ネガティブなフィードバックの有無** | 容姿・性格・主義思想への批判 |

度合いの計測しやすさ: `有無 > 数 > 長さ`。長さは面倒なので避ける。

### 計測方法とタイミング
正解はないが、習慣トラッカー / 自分で記録の二択。詳細は [習慣](habit.md) を参照。

タイミング: 一日の終わりに 1 回 / 数回（昼・退勤・就寝） / 気付いたとき随時。減らせば楽だが精度低下。

### 計測結果を踏まえてどうするか
- ポジティブ因子は **狙って増やせることはあまりない**。突発的なイベントが多い
- ネガティブ因子は減らす。発生源との対峙が必要

> リソースの管理には結局痛みを伴う行動が求められる

[盤外戦](bangaisen.md) と地続き。痛みを伴う行動が無理でも、**因子を把握しているだけでも意味がある**（後で対処のモチベが訪れる可能性）。

## 4: コンテキストの管理

ETM は一度の着手では終了しない。時間を空けて再開するときに前回の文脈を思い出す[コンテキストスイッチング](load_context.md)が消耗する。

> ETM を上手くやれるかどうかは、コンテキストスイッチングの消耗をいかに抑えるかにかかっている

ロードコンテキストの管理を本節では **コンテキストの管理** と呼ぶ（**コンテキスト** と略す）。

戦略は 2 つ:

| 戦略 | 内容 |
| --- | --- |
| **コンテキストメンテナンス** | 忘れない程度に高頻度で着手する |
| **コンテキスト駆動** | 着手を終わるときにコンテキストを書き残しておく |

### コンテキストメンテナンス
**Context Maintenance**。脳内のコンテキストを維持するために、とりあえず触れるだけの行為。

[着手ベース](habit.md#3-act仕掛けをトリガーにして行動する)（『先送り0』参照）の発展形で、**着手はゼロだが読み返した** も許容する。コンテキストを維持できれば良い。

長大な文書の作成（プレゼン資料・論文・レポート・書籍）と同じ。読み返していれば脳内のコンテキストは保持される。

### コンテキスト駆動
**Context Driven Work**。着手を終わるごとにコンテキストを書き残す。次回着手時にそこを読んで思い出す。

論点 3 つ:

| 論点 | 答え |
| --- | --- |
| **どの単位で書くか** | トピック単位（必要に応じて） |
| **何を書くか** | 「次に何をすればいいか」+「なぜこのトピックに取り組んでいるのか」+ 意思決定の変遷 |
| **どこに書くか** | 行頭・行末・進行中の場所など正解なし。すぐ探して読めることが重要。記号・絵文字（📒🧠）・タグ文字列で目立たせる |

> アンチパターン: 進捗を書く

進捗は管理 1.0 の概念。コンテキストに含めると駆動の営み自体が管理の色を帯びる。

### 同時にしない
両方使うのが望ましいが、**同じトピック内で両方を同時には行わない**。

> コンテキストメンテナンス中はコンテキストを書かない

書く・更新は読むより消耗する。コンテキストメンテナンス自体の腰が上がらなくなる。

不意に思いついた良い考えはコンテキストではなく **トピックのコンテンツ** として書く。

## 5: タスクの管理

[LTM におけるタスク管理](ltm.md#タスクに関する-3-種類の情報) に従う。タスクノート + タスクビュー + アンカバード。

ETM では 1〜4 が「やってもやらなくてもいい」を扱い、5 が「やると決まったこと」を扱う。
