# ブロックライティング (Block Writing)

[ETM](etm.md) におけるトピックの育て方の主要技法。**ブロック**（空行で区切られた箇条書きの塊）を並べることで本質を導く。

## 系譜

| 名称 | 内容 |
| --- | --- |
| **CBS**（Consideration Breakdown Structure） | シンプル版。検討ブロックを使い最後にまとめブロック |
| **サマリー駆動** | サマリーブロックをつくる → 仕切り直しを繰り返す |
| **ブロックライティング** | 一般化版。多種類のブロック + サイクル + レター |

## CBS

**Consideration Breakdown Structure**（検討のブレイクダウンと構造化）。シンプルな書き方。

```
(検討1)
 (検討1の内容)
 (検討1の内容)
 ...

(検討2)
 (検討2の内容)
 (検討2の内容)
 ...
```

### 書き方の規約

| 規約 | 内容 |
| --- | --- |
| 行指向 | 行単位で書く |
| インデントは **1-space** | 2-space 以上も可 |
| インデント配下は **緩い包含** | 厳密な論理構造は不要、主観で関係があると思ったものを束ねる |
| 空行で **検討ブロック** を区切る | 空行 2 行以上、`---` などで強調も可 |
| 順序性なし | 順序に意味があれば数字を振るなどで明示 |

### 検討（Consideration）
> タイトルのついた情報の塊

思考と言えないほどの羅列・思いつきも OK だが、**必ずタイトルは含める**。一時的な無題は許容するが、雑でも自分の言葉でタイトルをつける意識を持つ。

### まとめブロック
> どこかで大胆にまとめる

書き込みが増えると収拾がつかなくなる。要約 or 蒸留を **まとめブロックをつくる形で** 行い、以後そのまとめをインプットにして検討ブロックをゼロからつくる。

これが **サマリー駆動**。意思決定の言語化。これをしないと書き込み量に破綻する。

## サマリー駆動

要所要所でまとめブロックをつくる → それをインプットにして仕切り直す、を行う営み。

> 認知コストの限界とコンテキストスイッチングの壁を超えるために、面倒でもサマリーをつくって絞っていく必要がある

サマリー = チェックポイント / セーブポイント。ETM における最重要概念の一つ。これと同等のことができれば手法は何でも良い。

[コンテキスト駆動](etm_management.md#コンテキスト駆動) との関係: サマリーをつくるのが厳しい場合でも、最低限コンテキストメンテナンスやコンテキスト駆動を行うこと。

## 下から上に書く
通常テキストは上から下に書くが、CBS / ブロックライティングでは **逆方向も可能**。

```
(検討3) ← 新しい
 ...

(検討2)
 ...

(検討1) ← 古い
 ...
```

メリット: 書くときにスクロール不要、読むときに最新の状況から進める。

サイクルごとに方向が異なることもある（1 サイクル目は上から下、2 サイクル目は下から上、など）。読み取りが負担なら一言補足する。

> そもそも読む方向はあまり重要ではない

それよりサイクルの区切りとサマリー内容の方が大事。

## ブロックライティング

CBS の一般化。多種類のブロックを使う。

### ブロックレター
ブロックの種類を示すアルファベット。

> 指定は必須ではないが、あるのとないのとでは後で読む際の認知コストがかなり違う

### ABCD ブロック + Summary

| レター | 名称 | 役割 |
| --- | --- | --- |
| **a** | **Action**（タスクブロック） | 具体的に実践可能なタスク。✅ で完了 |
| **b** | **Brast**（ブレストブロック） | テーマ未定で何でも発散 |
| **c** | **Concept**（概念ブロック） | 新しい概念や用語の整理 |
| **d** | **Direction**（方向性ブロック） | 「こんな感じが良さそう」など方針 |
| **s** | **Summary**（サマリーブロック） | 「ここまでのまとめ」 |

### サイクル

> ブレストや検討で発散して、概念や方向性やタスクで肉付けしつつ、きりのいいところでサマリーをつくる

これを **1 サイクル** とし、何回でも繰り返せる。終わりはなく、どこで切り上げるかは自分次第。

> 次のサイクルを開始する際は、なるべく前回のサイクルのサマリーだけをインプットにする

### 検討ブロック (k)
ABCD ではないがよく使う。**特定のテーマ（お題）を定めて発散する**（ブレストはテーマ未定）。

| | 違い |
| --- | --- |
| **b ブレスト** | テーマ未定 |
| **k 検討** | 特定のテーマを定める |

両者の厳密な区別に意味はない。ブレスト中に深堀りに入る、検討中に発散になることもよくある。著者は検討ブロックを使うことが多い。

### その他のブロック

| レター | 名称 | 役割 |
| --- | --- | --- |
| **g** | **Glossary**（用語ブロック） | 以後使う用語の定義集 |
| **r** | **Reference**（参考情報ブロック） | URL / 書籍 / 人の発言など事実情報 |
| **cx** | **ConteXt**（文脈ブロック） | 開始時 cx0、サイクルごと cx1, cx2... |

#### 用語ブロック vs 概念ブロック

| | 内容 |
| --- | --- |
| **用語ブロック (g)** | 以後実際に使う言葉。読者も「このノート内で使われている」と捉える |
| **概念ブロック (c)** | 「こういう概念を定義したら便利」「まだ言語化できないが、こんな感じ」 |

使い分けは細かい話。サマリーに含めてしまっても良い。

#### 文脈ブロック (cx)
開始時の **cx0** がもっとも重要。

> 「このトピックを検討しているのはなぜなのか」「なぜ起きたのか」

cx0 を書いておくと、検討が煮詰まったときに原点に立ち返れる。やっぱり重要だからやる、要らないから捨てるなどの判断が可能。書いてないとギャンブル的な判断になり後悔の種。

cx1 以降はサイクルごと（1-サイクル 1-文脈ブロック）。サマリーに含めても良い。

[コンテキスト駆動](etm_management.md#コンテキスト駆動) のレターによる実装。

### 例

```
cx1
 色々検討始めてるけど、選択肢多すぎてなんかだるいなーってきもち

r 友達に聞いてみる、一言でまとめる
 友達1「...」
 友達2「...」

✅k 都会？田舎？
 徒歩で生活できるならどこでもいい
 徒歩片道20分くらいなら許容、チャリもあるし

k どんな家が欲しい？
 ネットは必須、動画見たいゲームしたい
 1K 25㎡は狭すぎ、もう一部屋広いの欲しい
 最上階、上の足音うるさいの本当にむり

cx0
 都内の家賃キツイので引っ越したい、郊外？
```

cx0 を読み返せば「問題の本質は家賃」と気付けて、家賃を下げる or 収入を上げる方向に絞れる。

## レターのカスタマイズ

> 必要に応じてレターをつくったり変えたりしても構わない

著者の例:
- 文脈ブロックは cx のみ（cx0/cx1 は使わず、開始時のみ。それ以外はサマリーへ）
- 実装ブロック (i, Implementation) — エンジニア向けに「具体的にどうつくるか」
- タイトルブロック (t, Title) — サイクルの名前

ただし複数人 ETM の場合は統一が必要。「レター一覧」トピックをつくってメンテする。

## チートシート

| カテゴリ | レター |
| --- | --- |
| **ABCD** | a / b / c / d |
| **サマリー** | s |
| **検討** | k |
| **拡張** | g（用語）/ r（参考情報）/ cx（文脈） |

ワークフロー（1 サイクル）:
1. b や k で発散
2. c や d で概念や方向性
3. a でタスクリスト（見えてきたら）
4. s でサマリー

サイクル間は前回のサマリーをインプットに。
