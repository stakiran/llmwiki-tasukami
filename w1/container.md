# コンテナ (Container)

タスクを分類したもの、または分類の各項目。[オルタスク](altask.md) のひとつ「入れ物」の本格定義。

書類をクリアファイル、本を本棚、ファイルをフォルダで分類するのと同様、タスクも色々な分類に分けられる。物理的というよりも論理的に意味を分けるニュアンスが強い。

分類は厳密でなくてもよい。むしろタスク管理は各個人各組織にあった分類を自分で設計するもので、雑な分け方も普通。

## よく使われる名前

| 名前 | ニュアンス |
| --- | --- |
| フォルダ | 階層的に分類（接続モデル寄り） |
| カテゴリー | きっちり分類（箱モデル） |
| グループ | 人を束ねる、カテゴリーと類義（箱モデル） |
| タグ | 性質の表明、複数つけられる（接続モデル） |
| ラベル | タグ + 色情報（接続モデル、リッチ） |
| プロジェクト | きりよくまとまった一連のタスク。多義的 |
| ワークスペース | 作業場、リッチコンテナの典型 |

各用語の詳細は本ページ後半。

## リッチコンテナ / プレインコンテナ

| 種類 | 内容 | 例 |
| --- | --- | --- |
| **プレインコンテナ** | タスクのみを分類 | フォルダ、カテゴリー、タグ |
| **リッチコンテナ** | タスクの分類 + 他の情報や機能 | グループ、ラベル、ワークスペース、プロジェクト |

ワークスペースはメンバー情報やアップロードファイルも持つ。ラベルは色情報を持つ（一応リッチ扱い）。

## 箱モデル と 接続モデル

| モデル | 特徴 | 例 |
| --- | --- | --- |
| **箱モデル** | ある対象を同時に一つの箱にだけ入れる | フォルダ、カテゴリー |
| **接続モデル** | ある対象に目印をつける（複数 OK） | タグ、ラベル |

- 箱モデル: シンプル。ただし「どっちにも該当する」**こうもり問題** が起きる。図書館の日本十進分類法レベルの分類を個人で確立するのはほぼ不可能（人類が挑むものではないとさえ著者は感じる）
- 接続モデル: 柔軟性は高い。つけすぎると形骸化

## コンテナはタスクとして扱わない

> 仮にタスクとして扱ったとしても十中八九うまくいかない

コンテナは粒度の粗い事柄なので、具体的に何をすればいいかがわからない・決断できない。タスクは粒度の細かいものであり、明確だからこそ管理できる。

> コンテナとは粒度の粗いタスク

ただし、粗いがゆえにタスクとして扱うのは難しいから分類として扱う。

## 扱い方の 4 段階

1. ツールのコンテナ機能を使うし、ツールの通りに使う（**設定より規約** の世界。例: GTD）
2. ツールのコンテナ機能を使うが、独自に使う（多数派）
3. ツールのコンテナ機能を使わないが、何らかのコンテナは使う（カレンダー重用者など）
4. 何も使わない（実は [一箇所にまとめる](strategies.md) 戦略は 1 個のコンテナを使っているとも言える）

## それはタスクか、コンテナか

人次第、状況次第。粒度をどう捉えるか。

著者の捉え方:

- **タスク** = 具体的・実行可能・始めれば数日以内で終了・終わったかはっきりわかる
- **コンテナ** = それ以上の粒度、複数のタスクを束ねる単位

例:

```
📦引っ越し
├─ ❓スケジュール組む
│  └─ 📝試す
├─ 📦引越会社の選定
│  ├─ 📝リストアップ
│  ├─ 📝見積もり依頼 A社
│  └─ 📝選ぶ
├─ 📦荷造り
├─ 📦廃棄
└─ 📦新しい家を探す
```

属性や性質で分類したいときもコンテナ化（タグ化）する。「調べ物」というタスクをつくるのではなく、「調べ物」タグをつけて関連タスクを後から辿れるようにする。

## 分類のメリット = コストの軽減

- **探すコスト** — 同じ種類のものを集めれば探しやすい。雑な分類でも構わない
- **切り替えるコスト** — 同じものをまとめれば「同じ頭の使い方で扱える」。違うものが混ざっていると認知的に苦労（コンテキストスイッチング）

## 自分でつくる

タスク管理の分類は基本的に自分でつくるもの。事例やパターンを参考にしつつ、自分に合うものを設計する。

> 階層の思想は強いが、別に無理して従う必要はない

階層に抗う例: **Scrapbox**。ページとページのリンクで Wiki を成立させる、ネットワーク構造のあり方。

> 分類が絶対的ではないこと、自分なりにやればいいこと、何ならやらなくてもいいこと

分類作業が目的になり、適応できない自分を責めたり維持に苦労したりするのはあるある。

## 各用語の詳細

### フォルダ
ファイルやノートの分類で主流。タスク管理ではあまり多くないが Wrike などが採用。**階層的に分類するニュアンス**。本質的にはタグに近いが「サブフォルダ」など階層性を持たせる。

### カテゴリー
コンテナとしてよく登場する。**箱モデル** で 1 カテゴリー 1 タスクを守ってきっちり分類。階層性を持つ場合と持たない場合がある。

### グループ
**人を束ねる** ニュアンスが強い。タスク管理での例は少ない（Yahoo!カレンダー等）。意味的にはカテゴリーと同義（箱モデル）。

### タグ
**性質の表明**。1 タスクに複数つけられる接続モデル。雑運用で何百ものタグが乱立しがち。X のハッシュタグも同じ感覚（性質表明）。**分類というより性質の表明** と捉えるのがわかりやすい。

### ラベル
タグ + 色情報。GitHub Issues など。視認性は上がるが乱用するとちかちかする。

### プロジェクト
**多義語**。すぐには終了しない、ある程度きりよくまとまった一連のタスクの集まり。生活なら「引っ越し」レベル。ビジネスでは数年継続する大規模事業も指す。

メソッド・体系で含意が違う:

- **PMBOK** — 一時的（永続せず）かつ独自の成果。ルーチン作業はプロジェクトではない
- **GTD** — PMBOK と定義が違う

タスク管理として捉えるなら「活動」「事業」程度の理解で OK。ただしツール背景を知っておくと便利。

### ワークスペース
作業場・作業空間。**典型的なリッチコンテナ**。タスク以外の情報も色々まとめる。場所のニュアンス。通常 ワークスペース → プロジェクト → タスク の階層。

## 分類対象と分類項目

### 大中小タスク
階層的にタスクを分解。3 段か多くて 4 段が良い（深すぎると全体把握困難）。

各階層のニュアンスを固定するのが重要:

- 大タスク = 方向性
- 中タスク = 目標
- 小タスク = 一日以内に終われる具体的なもの

**PARA メソッド** は 4 段階（**4 Based**）。人間のワーキングメモリ含めた研究結果に基づくとされる。

### タスクと情報の仕分け
受信箱には情報とタスクが混じる（タスクの方が少ない）。タスク側を特別扱い:

- **箱モデル** — 「タスク」フォルダに入れる
- **接続モデル** — タスクであることを示すタグ・マークをつける

> タスクだとわかった時点ですぐにコンテナに入れる（つける）

あとでまとめて行うのは苦しくて形骸化しがち。チェックする時間（朝/昼/晩 3 回など）を設けて一気に仕分けるのが現実解。

専用ツール: **HEY**（仕分けを明示的に強化したメーラー）。

### コンテキストタグ
「いつ実行できるのか」「どんな性質なのか」を示す用のタグ。詳細は [コンテキストタグ](context_tag.md)。

### セクション
時間帯（[Slotter](slotter.md) 参照）。人はどの時間帯に何をするかが大体決まっているので、セクションをコンテナと捉えてタスクを入れる。著者は午前セクションを「1: 起床後〜出勤」「2: 出勤後〜皆出勤前」「3: 皆出勤後の午前残り」に分割して、1 と 2 で集中業務を割り当てている。

## コンテナのパピプペポ

コンテナとしてありがちな使い方をパピプペポで整理:

| 文字 | カナ | 英語 | 意味 |
| --- | --- | --- | --- |
| **パ** | パーパス | Purpose | 目的・目標。大タスクや中タスク |
| **ピ** | ピープル | People | 人。誰に何を任せる/言うか |
| **プ** | プレイグラウンド | Playground | 遊び場。やりたいことを雑多に。おもちゃ箱 |
| **ペ** | ペンディング | Pending | 待ち。GTD でいう「連絡待ち」 |
| **ポ** | ポスト | Post | 受信箱と送信用下書き箱 |

ピープルは半ばこじつけだが、人をコンテナと見なす発想 = **コミュニケーション 1.0**（「常に特定の具体的な誰かを想定する」メンタルモデル）と親和性が高い。情報共有が苦手な人は 1.0 にとらわれている可能性。
