# タスクソース (Task Source)

タスクを生み出す何らかのもの。特に言語化されたもの。Source = 源泉。

[解釈の公理](axioms.md) で述べたように、ある事柄がタスクかどうかは人次第。タスクソースもまさに解釈が絡むもので、ソースからいつどんなタスクをつくるかはその人次第。

例: 「引っ越ししたい」は遠い願望なら当面タスク化されないが、近所トラブルで喫緊に変わればすぐタスク化される。

> ソースはソースとしてリストアップなり掲示なりしておいて、これらから必要に応じて必要なタスクをその都度つくる

## 4 種類のタスクソース

| 種類 | 解説 | 例 |
| --- | --- | --- |
| **目標事項** | 達成可否が存在する、道しるべとなるもの | 目的、目標、大タスク、プロジェクトなどの[コンテナ](container.md) |
| **維持事項** | 恒久的に維持したいこと | 日課、習慣、一部の[モットー](motto.md) |
| **連想事項** | 何らかの事物を思い出す・思い起こすためのヒントとして使うもの | トリガーリスト、ブレストなどの発想法出力、何らかの俯瞰全般 |
| **発想事項** | ひらめきや思いつきを促すもの、インスピレーション源 | 人によって著しく異なる |

すべて言語化されたものを想定。具体的な人・場所・[物タスク](object_task.md)・画像・動画は本章では扱わない。

## 目標事項と維持事項の運用

[メモのフロー](memo.md) と同じ流れ: 定義 → 読み返し → タスク化。

### 1. 定義
**最も大事**。目標は達成まで続き、維持事項は一生続く。納得感または信念がないと続かない。

[アスリートタイプ](athlete_applier.md) なら「とりあえず半年続けてみるか」もできるが少数派。納得感も信念もない場合は最初から諦めた方がいい。手に入らない場合は誰かや場所に従ってとりあえず始めてみる手もある（依存しすぎ注意）。

### 2. 読み返し
日々ソースを見て現状をかえりみ、次は何しようかと考える。

- 目標事項は **進捗** で 100% に近づく可視化ができればモチベ維持しやすい
- 維持事項はそうもいかず、日々の読み返しやタスクを楽しめるかが争点

### 3. タスク化
毎回行う必要はないが、毎回判断するのも面倒で形骸化に繋がる。「毎日 2km 走る」のように予定化しても苦しい。**微調整という営みを避けると十中八九形骸化**。

> 微調整する努力ができるかどうかは一種の目安

GTD と PARA メソッドはこの一連の流れを言語化・体系化している（GTD では **レビュー** と呼ぶ）。

### 自己理解の重要性
納得感や信念を足らせるためには **自己理解**（自己認識・自己分析）が不可欠。これがないと自己判断もできず、言いなりか多忙で誤魔化すしかなくなる。

## 連想事項の運用

3 つの方向性:

1. **連想リスト**
2. **[発散と収束](divergent_convergent.md)**
3. **俯瞰のつくりこみ**

### 連想リスト（トリガーリスト）
連想を引き出すための自問自答集・ヒント集。チェックリストと違って全項目消化する必要はない。雑に使う。

例（生活）:

- **やりたいことリスト** — いつかやりたいこと。GTD の「いつかやるリスト」。数百〜数千件もよくある。重複・肥大化を気にせず追記、眺めるときは肩の力を抜く
- **人生について考えるときに見るリスト** — 書籍『28 歳からのリアル』の 8 要素（仕事/結婚/お金/住まい/健康/親/趣味/常識）
- **生活のコツ 128** — 明治時代の「簡易生活」前提の生活コツ集
- **自己啓発リスト** — Wikipedia の自己啓発主要テーマ

例（仕事）:

- **プロジェクトリスト** — 抱えている仕事や用事のリスト
- **ピープルリスト / ポジションリスト** — 普段絡む人や役割
- **オズボーンのチェックリスト** — 転用/応用/変更/拡大/縮小/代用/再配置/逆転/結合

ChatGPT に頼んで洗い出してもらうこともできる。

#### 連想リストと付き合うコツ

- **雑でもいいので使ってみる体験をする** — これができるかどうかがすべて。所詮箇条書きなのでわかった気になりやすく形骸化もしやすい
- **自分でカスタマイズ・メンテナンスする** — 合う合わないが激しい
- **わからない単語をちゃんと調べる** — 目に入る度に無視するノイズが積もると馬鹿にならない
    > 連想に頼るのが苦手な人の大半はノイズに潰されているからではないか

### 俯瞰のつくりこみ
**俯瞰（Overview）** = 全体または特定範囲を眺めること。眺められる状態になっていないので意識的に **俯瞰物** をつくる。

例: 仕事の登場人物を考えたい → 名前リスト + 顔写真の列挙。

俯瞰のコツ:

- 俯瞰そのもののコツはない。連想を引き出すだけ
- 大事なのは **俯瞰物を自分が使いやすいようにつくる** こと
- 既存の俯瞰物（プロジェクト概要資料、体制図、タスク管理ツールの[ビュー](views.md)）をそのまま使うのはアンチパターン
- アナログ・デジタル問わず。一般論として空間配置に長けた手段が良いが、人による

俯瞰物をつくる営みは面倒くさいが、慣れてしまった方が早い。

## 発想事項の運用

「言語的につくられたインスピレーション元」をいかに見つけ、付き合うか。**非常に個人的かつ再現性もない** ので解説は割愛（著者の例: デジタルツールのヘルプ、リラックスして読める仕事術投稿など）。

> 強張らず疲れてもいない、リラックスした頭でニュートラルに読めるかどうかが肝心

ひらめきは見逃すのでタスク化する（前段で[メモ](memo.md)化することも多い）。

## 締切との向き合い方: ソースアプローチとプランアプローチ

| アプローチ | 性質 |
| --- | --- |
| **ソースアプローチ** | タスクソースから適宜タスク化。主観に基づいて行えるが、締切と相性が悪い。**探索的** |
| **プランアプローチ** | 計画やスケジュールを組んでそのとおりに動く。仕事は基本これだが疲弊と二重苦になりがち。**計画的** |

> 現代の仕事はプランアプローチに支配されているが、これは管理上都合が良いから。方法として優れているからではない

仕事のプランアプローチが過酷なのは「計画はそもそもそのとおりに踏襲することが現実的に難しい」+「計画ベースの管理に従い続ける疲弊」のダブルコンボ。

最も汎用的な使い分けは **最初はソースアプローチ、どこかでプランアプローチに切り替える**。切り替えポイントが [SP 臨界点](sp_threshold.md)。
