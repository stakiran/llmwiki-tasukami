# リソースの管理

時間やお金を管理すること。[5 つの管理対象](management_targets.md) の 1 つ。

タスク管理では「消費量を記録・申告させる」形で現れる:
- タスクページに「かかった時間」を書く
- 月次で時間記録を提出
- お金を使う前に承認依頼
- 残業の事前申請

## 直接関係がないシャドー扱いされがち

リソースの管理は **インダイレクトシャドー** とみなされやすい:

- 各個人が使うものなので本人が把握・計上することになりがち
- ただしビジネスのコンプライアンスや個人の生死に関わるので無闇に削れない
- 「必要だから」と正当化されやすい

## 対処

- **タスク化** — 申請（お金、道具、設備）はわかりやすい。スポット系は事前にプール
- **バッファ・スラック** — [バッファとスラック](buffer_slack.md) で個別に吸収
- **記録会・申告会** — 打ち合わせの一種として「リソースの記録や申告を行う場」を設ける。手を止めて作業に集中、メンバー同士で「あれ忘れてない?」と補完
- **専任化** — リソースマネージャー / リソースウォッチャー。業務は苦手だがフォローや雑務が好きな人が適任
- **自動化・省力化** — 記録は自動化したい。手間を最小化。DX の出番

社内システムや経理部があるから大丈夫、ではない。**個人やチーム単位** でリソース管理がシャドー化するのを防ぐ話。

## 管理自体を減らす工夫

### 時間
バカ真面目な記録は本当に必要か再考。プロセス変更権限が無くても運用面で工夫できる。表向きは言えないので裏でうまくやる。逆に管理者が邪悪だと悪用（サービス残業を強いる）される。

### お金
申告プロセスを減らせる:

- **誰でも無条件に使えるが、誰がいつ何のために何円使ったかは全部見える（透明性）**
- **各社員・各チームに自由予算を支給／立替精算化** — 有償ツール、特に SaaS で重宝
- **承認者をランダムにする** — 金額に応じて n 人、チームメンバーから抽選。管理者ボトルネック解消

**Rate Limit**（使える上限を設定）と併用するとなお効果的。銀行の上限引き出し額、ChatGPT の API 課金上限など。

## まとめ
- リソース管理は記録・申告という形でシャドー化しやすい
- 対処はシャドータスク全般の応用 + 記録会／申告会
- そもそも管理自体を減らす工夫が色々ある
