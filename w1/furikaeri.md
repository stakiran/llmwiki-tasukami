# 振り返り

過去の情報をインプットにして考察を行い、アウトプットを出すこと。タスク管理が「戦略をなぞる」ことだとすれば、振り返りは「戦略を考える」もの。

## 定義

| 段階 | 内容 |
| --- | --- |
| **インプット** | 過去の情報（活動、行動、言動、定量データ、日記など） |
| **考察** | 自分の頭で考えて決める。必要に応じて主観以外の情報にも頼る |
| **アウトプット** | タスク + モットー（できればコンテキストも） |

> アウトプットを出すこと

考察だけでは「学び・気づき・よかった/悪かった」を挙げただけで終わる。次に繋げるためには **タスク**（次やる行動）か **[モットー](motto.md)**（次から心がけたい方針）を出す。コンテキストは努力目標で、未来の自分が思い出すための文脈。

> 振り返りに正解はありません

## PROC（4 つの検討要素）

| 英語 | 意味 | 解説 |
| --- | --- | --- |
| **P**urpose | 目的 | なぜ振り返るか |
| **R**emembering | 想起 | インプットをどうやって集める/思い出すか |
| **O**pportunity | 機会 | いつ振り返るか |
| **C**ontinuation | 継続 | 一度で終わらせるか、何度も続けるか |

> 一度の開催で無理に終わらせる必要はない

## DWMY と DWMY ヒエラルキー

定期的に行うのが鉄板。単位は **DWMY**: Day / Week / Month / Year。

問題: Month や Year は範囲が広くてインプット収集が困難。

解決: **DWMY ヒエラルキー** — 一段下のアウトプットをインプットにする。

| 段階 | インプット |
| --- | --- |
| Day | 日々の記録 |
| Week | Day の Output 7 個 |
| Month | Week の Output 4〜5 個 |
| Year | Month の Output 12 個 |

毎日律儀にレビューする手間とのトレードオフ。著者は GTD のように毎日・毎週・毎月レビューを全部行う。

流派:

- **DWMY** — 全部行う、律儀な人向け
- **DM** — 毎日 + 毎月、ずぼら向き
- **DW** — 毎日 + 毎週、俯瞰不要
- **WM** — 毎週 + 毎月、日単位は細かすぎる人向き
- **MY** — 毎月 + 毎年、いきなり Month から始められる人向き

## 振り返りは予定として扱う

タスクではなく予定として扱った方が上手くいく。集中してじっくり取り組む必要があるから。

> 振り返りを予定として確保することすらできない人は、スタートラインに立てていない

### 外部の力を借りる
ひとりで予定確保できない場合:

- **集中しやすい場所に出かける** — カフェ、コワーキング、人気の少ない公園、トリッキーにはカラオケやラブホテル（防音性、広さ、料金）。著者は自然の音を BGM に近所の山道へ
- **もくもく振り返り会** — pha さん発祥の **もくもく会** は **突発型の自習室**。各自もくもくと作業（私語厳禁）。振り返り限定で開催 or 参加するのも可。親しい間柄だと [ファミリア・デバフ](familiar_debuff.md) で私語厳禁が守れないので赤の他人を集めるのが◎

## 3 つの目的: KAP

| 略 | 目的 | 内容 |
| --- | --- | --- |
| **K** | Know（知るため） | 単に知りたい。アウトプットなしでも OK、続くなら良し |
| **A** | Advance（進むため） | 前進したい。課題と現実のギャップを埋めるサイクル（PDCA, OODA） |
| **P** | Protect（守るため） | Minimize Negatives。健康診断、忘迷怠の防止 |

## 4R マトリックス

|  | 自分に関して | 他人や活動・仕組みに関して |
| --- | --- | --- |
| **客観重視** | 1 **Reassessment** | 2 **Review** |
| **主観重視** | 3 **Reflection** | 4 **Recollection** |

- **Reassessment** — 自分のデータ（健康、お金）。事前のデータ整備が要る
- **Review** — チームやプロジェクト。データドリブン、エビデンス。仕事ではここが標準だが個人では大変なので 4 になりがち
- **Reflection** — 内省。自分と深く向き合う
- **Recollection** — 他人や活動を主観で。仮説検証が必須（決めつけにならないよう）

## 振り返りの設計やり方

### Enum and Assign 方式
1. 何の振り返りを行うかを洗い出す
2. 各対象について DWMY を意識した予定を取る
3. インプット・考察・アウトプットを設計

メリット: わかりやすい / デメリット: 大変

### 振り返りアワー方式
事前に時間を取り、当日その都度何を振り返るかを決める。シンプル、軽く試したいとき。**スタートラインに立つ試金石** にもなる。

### Diary Based DWMY
毎日日記を書く + DWMY ヒエラルキーで振り返る。日記はこまめに（一日の終わりだけだと思い出せない、それ自体が振り返りになる）。アウトプットは「次やりたいこと」と「次から心がけたいこと」が定番。

> まずは「次やりたいこと」を一つだけ、くらいにハードルを落とす

### 5 分振り返りんぐ
きりのいいタイミングで突然 5 分の振り返り。**大げさなロールプレイ** で一人二役以上を演じて会話する。

> プログラミングのテディベア効果（Bear, ベアプログラミング）— 困ったらぬいぐるみに相談する

演じることで脳の使い方が変わって新しい気づきが出る。

## ヒント集

### インプットの 3 やり方
1. **記録を見る** — 計測値や日記。事前の仕込みが要る
2. **記憶を思い出す** — 信用できないが直近のこびりつきには
3. **ヒントを借りて思い出す** — 外部の刺激や問いかけ

ヒントには **動的**（その場の刺激、外食しながら通行人を眺める等）と **静的**（事前用意のリスト）がある。基本は静的、たまに動的でバランス。

### 考察時のヒント

#### フレームワーク
チーム向けが多いが個人でも参考に:

- **KPT** — Keep / Problem / Try
- **YWT** — Yatta / Wakatta / Tsugiyaru（個人向きで雑にやれる）
- **mad, sad, glad / WRAP の A** — 感情系
- **リーンコーヒー / マリオカート** — 議題と論点の可視化

#### トリガーリスト
[連想リスト](task_source.md) と同じ。GTD のもとで「気になることを外に出すための自問自答集」。

大カテゴリ: 仕事 / プライベート / 趣味 / お金 / 健康 / 家族 / 住まい / 結婚 / 常識。

各項目を **MSCW**（Must / Should / Could / Wouldn't）で分けると、何のための振り返りかも整理しやすい。

## アウトプットのやり方

### タスクを出す
- **即席タスク**（一回限り）と **[ルーチンタスク](routine_task.md)** を使い分け
- 即席はリストアップだけだと先送りされるので予定に仕込むか初動コンタクトを送る
- 続けてみないとわからないものは **ルーチンタスク** に

### モットーを出す
- 限定的なモットー（仕事の文脈で「A さんに高頻度で声掛けよう」等）
- 人生全般のモットー（リフレクション系の振り返りで導出。エモットー → イモットーの作り変え）

詳細は [モットー](motto.md)。

### コンテキストを出す（努力目標）
ロードコンテキストを残しておくと、次の振り返り時に思い出せて [コンテキストスイッチング](load_context.md) の負担が減る。

> 未来の自分という他人 に引き継ぐつもりで臨む

辛くて続かないくらいなら出さなくていい。続いてからで遅くない。

## タスクデット (Task Debt)

> アウトプットしたタスクは、次の振り返りまでに完了させておくか、少なくとも一度以上着手して何らかの進捗や気づきを得ておく

これができないと **タスクデット**（負債）が積み上がる。TODO リストの膨れ上がり現象と同じ。

> タスクデットが積み上がらないバランスを心がけてください

積み上がる場合の対処:

- アウトプット量を減らす
- 振り返りの頻度を落とす
- 考察が甘くてタスクが大きすぎる → スモールスタートで小さなタスクに
