# LTM（文芸的タスク管理）

**Literate Task Management**。文章を書く営みをベースとしたタスク管理。

文芸的 = 「文章を書く営み」（芸術性は不要）。日記・日誌・備忘録のような **自分用の文章** を恒常的に書きながら、その中でタスクも扱う。

## 概要

| | 内容 |
| --- | --- |
| **メリット** | 文脈の尊重 |
| **目的** | QoL の向上 |
| **手段** | [ネットワーク型ノートツール](network_note.md) + ネットワークベースのスタイル |
| **誰向き** | ひとりで行う人。1 日 1 時間以上の読み書きの余裕がある人。デジタル前提 |

### メリット: 文脈の尊重
文章を書くことで人生・生活・仕事といった [文脈](context.md) がクリアになる。これを読みつつ、書き足したり書き換えたりしてタスクを考えることで、より文脈を尊重したタスクを扱える。

### 目的: QoL の向上
> 必ずしも生産性や成長に繋がらない

生産性や成長を求めるなら文脈を無視して必要なタスクを気合いで消化するのみ。LTM は自分の文脈を尊重することを第一にする。結果として向上するのは QoL。

慌ただしい場面の打開には向かない。

### ひとりで行う、人を選ぶ
個人タスク管理の手段。プロジェクト/パートナータスク管理は想定外。

不向きな人: 文章の読み書きを継続できない / 1 日 1 時間以上の余裕がない / アナログ派。逆に言語化能力やスピードは気にしなくて良い（タスク管理は本質的に個人的なもの）。

## LTM のスタイル

| スタイル | 内容 | LTM としての強さ |
| --- | --- | --- |
| **ダイアリーベース** | 日単位でテキストを書く | 弱い（[PTTM](pttm.md) を脱せない） |
| **プロジェクトベース** | 1-プロジェクト 1-ノート | 中（整理整頓ができる人向き） |
| **ネットワークベース** | 1-話題 1-ノート、リンクで繋ぐ | 強い（推奨） |

> LTM ≒ ネットワーク

ネットワークベースは [トピック指向](network_note.md#トピック指向) を採用し、プロジェクトベースとダイアリーベースも包含できる。

ダイアリーベースの弱点: 書いたテキストをひたすら読み返すコストが大きく、メンテナンスが手作業すぎる。プロジェクトベースの弱点: プロジェクト横断のメモの行き場がない、整理整頓が要求される。

## ホーム

LTM の拠点。よく見られるのは 4 種類。

| 種類 | 内容 | 向く人 |
| --- | --- | --- |
| **ホームノート** | 唯一の玄関口ノート | 自分で整理整頓できる人 |
| **ホームビュー** | ノートを俯瞰するビュー | Scrapbox など優秀なビューを持つツール利用者 |
| **日付ノート** | 日単位で日付名のノートを切る | 散らかりがちな人 |
| **トランク** | 多数のリンクを伸ばしたノート（複数可・遊牧的） | 状況に応じて拠点を切り替えたい人 |

### ホームノート
ホームに直接書き散らす + ある程度まとめる + リンク集の 3 構造が自然に生まれる。リンクが多くなりすぎたら別ノート（例「もう見てない」）に逃がす。

### ホームビュー
ノート一覧、検索ボックス、ソート/フィルタ、新規作成機能を備えたビューをホームにする。ノート自体はビュー上では書けないが、俯瞰ができる。Scrapbox の例。

### 日付ノート
今日 = `2024-08-06`、明日 = `2024-08-07`。[Dailer のデイリーエリア](dailer.md) と同じ概念。

ジレンマ: よく使うリンク集やデイリータスクリストを毎日コピーするのは面倒、別ノート（倉庫）に逃すと行き来が発生して **タスクが漏れる**。

> ホームとして常に表示し、常に見るノートは一つだけに留めたい

実用ハイブリッド: ホームノートに当月分の日付ノートへのリンク集を置く / Scrapbox の場合は今日の日付ページをピン留め。

### トランク
直訳「幹」。多数のリンクを伸ばしたノート。

> トランクの使い方は定住的でなくても良い

遊牧民のように `テスト` ノートから `岡山新生活` ノートへ、さらに `2024 旅行` ノートへ──と拠点を転々とする運用も OK。**別れの儀式 = リンクをつけて辿れるようにしておく**。

## タスクに関する 3 種類の情報

LTM では情報を 3 種類に分けて捉える。

| 種類 | 内容 |
| --- | --- |
| **メモ** | あとで処理する事柄を素早く一時的に残す。詳細は [メモ](memo.md) |
| **タスク** | やると決まった/決めたいもの。**タスクビュー** + **タスクノート** の両輪 |
| **文脈** | メモでもタスクでもない、ノートに書いたことすべて |

### タスクビュー / タスクノート
- **タスクビュー** — 複数のタスクを集めて俯瞰する見せ方。GTD のネクストアクションのように一箇所に集める手も、ノートごとにタスクリストエリアを分ける手もある
- **タスクノート** — タスク T それ自体に関するノート。T に関する文脈（属性・関係・機能）を集める/集める

> LTM ではタスクノートでタスクの文脈をネットワーク的につくりつつ、タスクビューでも俯瞰する両輪の戦略

通常の「ザ・管理的なやり方」はしない。それはタスク管理であって文芸的タスク管理ではない。

## エリアとマーク

メモとタスクは「後で処理する」必要があり、本質は **読み返し**。読み返しの労力を減らす戦略は 2 つしかない。

| 戦略 | やり方 |
| --- | --- |
| **エリア** | あとで読み返す場所を決めて、そこに書く |
| **マーク** | 読み返したい部分にマークをつけて、検索でビュー化 |

### エリア
ホームに「デイリータスクリスト」エリアを設けたり、別ノート丸ごとをエリアにしたり。

> 重要なのは書き込みのしやすさ、そして後でちゃんと読み返せるかどうか

慣れすぎてスルーする恐れあり（[ネグレクテッド](dousen.md)）。カレンダーやリマインドで定期的に強制チェック、レビュー時に「読み返すエリア一覧」を全部読む等。

LTM では恒常的にノートを読み書きするので、土地勘のように定着すれば数ホップで到達できるようになる。

### マーク
読み返したい部分に印（記号や絵文字）をつけて、検索でビュー化する。

例: `@task` / `●task` / `■task` / 絵文字（📒 ✅ 🏃 📌 💣）

完了マークは別に決める。例: `💣タスク1` → 完了時 `✅💣タスク1` で重ねる。

> 検索結果を開く手間はできるだけ抑える

ショートカット、検索結果のブックマーク、自作スクリプト等で工夫。

応用: マーク以外にタスクの[属性](attributes.md)も書いておく。例: [hown](https://github.com/stakiran/hown) は `[2002-10-22]@ タスク` のように [日付系属性](attributes.md#日付系属性) を含めて、現在日との比較で優先順位を制御する。文芸的タスク管理の上手い例。

## アンカバード（Uncovered）

LTM でカバーできない部分。

| 種類 | 対処 |
| --- | --- |
| [ルーチンタスク](routine_task.md) / [オルタスク](altask.md) | [PTTM の扱わないもの](pttm.md#扱わないもの) と同じ。専用ツールへ |
| タスクや文脈を他者に見せる | LTM は [管理的安全性](pttm.md#3-誰にも見せない管理的安全性) 前提。他人に見せる用途は推奨しない |
| 細かい属性管理・高機能ビュー | 通常のタスク管理ツール |
| 締切厳守の厳しめ管理 | 通常のタスク管理ツール / カレンダー / リマインダー / イベント |

> アンカバードのやり方ばかりに頼っていては LTM はできない

なるべく LTM で完結し、面倒な部分のみ別の手段に頼る。料理や DIY と同じで「市販調味料は使うが料理は自分でする」感覚。

## 文脈

### 捉え方: ヒント
LTM では「やらないといけないものを上から処理する」のではなく、**自分主導** で日頃扱っている文章からタスクを決める。

> 文脈とはヒント

正解はない。航海日誌のような客観記録、感情の主観記録、書籍からの引用と考察、深い思索 — 何でも文脈になる。

### つくりかた: 3 ポイント

| ポイント | 内容 |
| --- | --- |
| **書きたいことを書く** | 続かなければ話にならない。日記、趣味、愚痴、何でも |
| **深堀りできるネタを捉える** | 直近のタスク、やりたいこと、[トリガーリスト](furikaeri.md#トリガーリスト)、SNS/ニュース/書籍のキャッチアップ。「美味しいネタ」が自分の軸になる |
| **自問自答** | 自分はどう思うか / 行動に繋げるなら何ができそうか / なぜそう感じるのか / 建前と本心 / 続きをやるならどう etc. |

[GTD の収集ステップ](exploration.md) のように 1〜2 時間確保して書き殴るのも有効（**探索**）。

## ライティングスタック

| スタック | 原本の場所 |
| --- | --- |
| **イメージスタック** | 頭の中 |
| **ライティングスタック** | ノート |

LTM ではノートを原本にする。頭の中はノートを読み込んで考えるための作業場にすぎない。哲学が変わったらノートに反映する。

> LTM が定着するかどうかは、ライティングスタックの考え方に染まれるかどうかにかかっている

タスク管理の主な恩恵「脳内ではなくツール側に保存して脳内をスッキリさせる」を、LTM ではノートで実現する。然るべきノートを然るべきタイミングで見ることで思い出せる/切り替えられる。

## 切り出し

有益と思われる文章や単語を、それ自体をノート名にして独立ノート化すること。

例: 引用文中の「LTM を行う者にとって書いたノートが原本」を `[ ]` や `[[ ]]` で囲んでリンク化 → 同名のノートが新規作成される。

> LTM では文章を何でも部品化できる

作家がことわざを使い回すように、プログラマーがスニペットを部品化するように、LTM では切り出しで文章を部品化する。後々別のノートで「これノートの方を原本にするってことだよな」と気づいたら、その切り出しノートへリンクを書き足せる。被リンクが貯まれば自分にとって重要なノートだとわかる。

> 切り出しが文脈をつくる

## 微修正とエイリアス

切り出したノート名は **わかりやすく / キャッチー** に修正する。リンク化はノート名がベースになるため、扱いやすさが再読解・再利用の頻度を決める。

例: 修正前「LTM を行う者にとって書いたノートが原本」→ 修正後候補「ノートが原本」「文芸的タスク管理では書いたノートが原本」など。原意の尊重・厳密性は不要。**自分にとってわかりやすいかどうか** がすべて。

### エイリアス
複数の表現で迷う場合、全部つくる。「ノートが原本」「文芸的タスク管理では書いたノートが原本」「LTM を行う者にとって書いたノートが原本」を全部用意して、どれからも同じノートにアクセスできるようにする。

ルール: **1 ソース n エイリアス**。集める先（ソース）はただ一つ。無闇に増やさず、自然と分岐するときだけ使う。

## アイキャッチとエキサイトメント

ノートが貯まると活字の海になる。継続的に向き合うモチベーションを保つ仕掛けが要る。

### アイキャッチ
キャッチーなノート名、画像、GIF、動画。視覚的なアクセントで活字の海を賑やかにする。タスク管理に直接関係なくても OK（SNS のバズ画像、VTuber 動画、自分のゲーム/ダンス動画など）。自己肯定感が上がる効果も。

ただし他人のコンテンツばかり集めると **ただのブックマーク展示場** になり、自分の文脈が育たない。

### エキサイトメント
作業興奮を引き起こす構成。**書きかけのノートを優先表示** する。

Scrapbox の Date Modified（更新順並び）やピン留めが代表的。やりかけ・中途半端なノートが先に表示されて手をつけやすい。

| 種類 | 内容 |
| --- | --- |
| **静的なエキサイトメント** | ピン留めで常に固定（3〜7 個程度に抑える） |
| **動的なエキサイトメント** | Date Modified 等の動的並び替え。定期/不定期に眺めて手を加える |

広く確保すると判断疲れで消耗する。**自分に合った広さ** が大事。

## 文芸的タスク管理の本質: 見えてきたらさっさと行動する

> 文芸的タスク管理は、一応タスク管理といいつつも、本質はノートテイキング

タスク管理は「タスクを決める段階」と「決めたタスクを消化する段階」に分かれる。LTM は **前者** にフォーカスする手法。

| 段階 | 適した手段 |
| --- | --- |
| タスクを決める | LTM（文脈を育てて見えてきたらタスク化） |
| 決めたタスクを消化する | 通常のタスク管理ツール |

決まったタスクは、LTM ではなく通常のタスク管理ツールで管理した方が上手くいくことが多い。

LTM は自律的・創造的な生き方を支える。慌ただしい状況下で瞬発的に次々こなす場面ではさすがに役に立たない。
