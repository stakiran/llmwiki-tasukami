# Alternate モデル

[Alternate（俯瞰と注視）](alternate.md) がどの段階で実現されているかを 4 段階で表したモデル。

| 段階 | 名前 | 実現範囲 | 例 |
| --- | --- | --- | --- |
| 1 | **N-Micro** | 注視のみ | タスクが各自の脳内にのみ存在 |
| 2 | **1-Macro** | 俯瞰のみ | メンバー全員が見る何とか票や何とかボード |
| 3 | **1-Macro N-Micro** | 俯瞰と注視 | 固定的なビューを持つタスク管理ツール |
| 4 | **N-Macro N-Micro** | 俯瞰と注視 | ビューをカスタマイズできるタスク管理ツール |

## N-Micro

注視のみ。最も単純な例は「各自の頭の中にのみタスクが存在する」状態。

- 私生活や趣味では基本的にこれ
- ビジネスでも少人数なら通用することがある（ティールに近い形）
- 労働環境の原始性との相関が高い
- 「タスク管理どうしてますか?」が就活・転職のキラークエスチョンになる

物理的または画面越しに進捗が見えるのも一種の注視（共有フォルダなど）。ただし間接的で、各自の脳内に依存することからは逃れられない（A さん「終わった」/B さん「いやまだ」が起きる）。

> N-Micro は、いわばメンバーの数（脳の数）だけ原本が存在するようなもの

## 1-Macro

俯瞰のみ。

- 単純例: 管理者の頭の中だけで俯瞰が組み立てられている。リモート可能な大企業でも頻出
- 管理者がボトルネックになり、上下関係が激しくなる
- 解消手段として WBS / ガントチャート / 何々管理票 / ボードといった俯瞰装置を使う

俯瞰装置の限界:

- 「よりすぐりのタスク」だけが載る
- それ以外（こぼれたタスク）は無視される
- 現実は仕事も人も多様で、こぼれが多いと誰かが無茶することになる
- 管理者は痛みを感じないし、契約や予算で固定されていて融通が利かない

> トップダウンの強さと固定的な視野からは逃れられない

## 1-Macro N-Micro

俯瞰と注視の両方をサポート。固定的なビューを持つタスク管理ツールを使う段階。

- タスクはツールに登録され、データから俯瞰ビューがつくられる
- タスクごとのページもあるので細かいフォロー可
- 記録が残るので記憶のいいかげんさが消える
- いわば **タスク管理に関する DX**

> タスクという情報を共通言語にしてコミュニケーションが取れる

メンバー同士で議論可能。意思決定者として管理者がいると便利だが、メンバーでできるなら管理者は要らない。

### プロジェクトタスク管理におけるキャズム

第 3 段階に至るには非常に高いハードルがある。これを **キャズム（深い溝）** と呼ぶ（イノベーター理論のキャズムから）。

- ツール導入とメンバーのスキル
- 形骸化させないよう日頃から入力し、これでコミュニケーションする
- 「全員に同じデジタルツールを使わせる」「常用させる」ことの難しさ

実際、第 3 段階のツールは主にクリエイティブな仕事をする人や組織で使われる。学習機会が多く、自分のペースで働く重要性を知っているから。

一般人の集団でキャズムを超えるには長い啓蒙＋ツール費用＋啓蒙役の人件費が必要 → 経営者の協力が不可欠（IPA の DX 白書 2021、マッキンゼーの緊急提言）。プロジェクト範囲では、その組織単位の経営者に相当する誰かの協力と投資が要る。

## N-Macro N-Micro

ビューをカスタマイズできるタスク管理ツールまたは仕組みを使う段階。

- 「こういう観点で見たい」要望に応えられる
- 例: monday.com Work Management のウィジェット。タスク属性のカスタマイズもでき、優先度 1-5 段階＋緊急属性の二段構えなどもできる
- 緊急属性が立ったら問答無用で着手、を一覧化するウィジェットもつくれる

### API

カスタマイズ手段としての **API**。「プログラムから呼び出されることを想定した、各機能各データへのアクセスの口」。

- 現代のデジタルツールは内部で API を組み合わせている。パーツ
- 公開すれば他者が使ってくれる、思わぬ使い方を実現してくれる
- ファンや存在感が増し、サステナブル
- 自前で API を使ってデータを取り、お望みの見せ方で見せる手も。専門的な開発能力と時間が要る
