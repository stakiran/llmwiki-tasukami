# PTTM（プレーンテキストタスク管理）

**Plain Text based Task Management**。プレーンテキストとエディタを用いてタスク管理を行うこと。

目的は **自由自在にタスク管理を行うこと**。アナログで紙とペンを使うように、デジタルでもタスクを自在に扱いたい。

## 背景

通常のタスク管理ツールは、ビジネスや多人数運用に通用させるために煩雑な画面・画面遷移・通信待ちを伴いがち。これは個人タスク管理にはオーバースペック。

> 自分用の、小回りの利くタスク管理を行うには

→ プレーンテキストをエディタで扱えば、テキストをどう書きどう回すかの部分で工夫ができる。

詳細な前提知識は [プレーンテキスト](plain_text.md) と [5 つの道具](plain_text.md#5-つの道具) を参照。

## PTTM ≒ タスクリストの運用

PTTM はタスクリストを運用するだけ。工夫として用いるのが [属性](attributes.md)。

### TODO リストの限界
最も有名な TODO リストは「名前」と「状態」だけのタスクリストで、すぐに破綻する。

```
ごみ捨て
新しい冷蔵庫を調べる
✅家電量販店
通販
✅上司が前買い替えた言ってたので話聞く
コバエうるさいので対処したい
```

何が終わって何が残っているか、いつやるか、どれが関連しているか――数日で混乱する。

### 属性で工夫する

[実行日属性](attributes.md#実行日-starting-date) を導入し、デイリータスクリストのルールを入れると:

```
■今日: 2024/07/29
ごみ捨て
新しい冷蔵庫を調べる
✅家電量販店
✅上司が前買い替えた言ってたので話聞く
通販

■2024/07/30

■2024/07/31
コバエうるさいので対処したい
```

時間軸の導入だけでも融通が利くようになる。先送りはタスクを別エリアに移すだけ。

### ファイル分割の工夫
過去ログを別ファイルへ:

```
[task.txt]      = 今日と明日以降
[tasks_old.txt] = 完了タスクの記録
```

3 ファイル構成にもできる:
```
[today.txt] / [tomorrow.txt] / [yesterday.txt]
```

ただしファイル間移動は面倒。1 ファイルで `■` `●` `★` などで区切る運用もアリ:

```
■================
■今日
■================
...

●================
●明日以降
●================
...

★================
★昨日以前
★================
...
```

### エッセンス

> 属性とファイルの分け方を工夫することで自分なりに運用していく

正解はない。**自分に合ったやり方は自分にしかわからず、模索していくこと** が PTTM。模索に耐えられない人は PTTM から回れ右する。

## 設計原則

PTTM を成立させるための 6 原則。

### 1: 管理範囲を決める
PTTM としてどこまでをタスク管理するかを定める。

選択肢は実質 2 つ:
- 特定の対象だけ管理（例: A 案件だけ）
- 生活のすべてを管理

著者おすすめは **後者**。個人タスク管理では複数の役割や文脈をまたいだ検討が必要なため。仕事とプライベートの境界が曖昧（ワークインライフ / ワークアズライフ）であるのと同様、PTTM もなるべくいっしょくたに扱う。機密性の関係で、多くの労働者は仕事用とプライベート用の 2 つの PTTM を回すことになる。

### 2: 動線化する
PTTM で使うファイルは、毎日頻繁に通る場所に置く。[動線](dousen.md) と [RSAF サイクル](rsaf.md) を取り入れる。

> 動線化が定着するかどうかが PTTM の成功の分かれ目

PTTM ではテキストを頻繁に読み書きする。面倒くさいと考えるのではなく、面倒と感じないほど手になじませる。

### 3: 誰にも見せない（管理的安全性）

> PTTM の中身はプライベートなものであるため、誰にも見せてはいけない

恥ずかしさのリスクを潰せている状態を **管理的安全性** と呼ぶ。極限まで確保すべき。

個人タスク管理が形骸化する主因は **「恥ずかしいから」**。誰かに見られる可能性があると、心理的にセーブがかかって **一部のタスクを書かなくなる**。

書かないタスクの例: アダルトな営みのタスク、セクシャルマイノリティ関連の手術、障害に関する通院や調査、現職や家庭に伏せている転職活動。

書くタスクと書かないタスクが混在するほど形骸化する。判断に認知コストを費やすうえ、面倒くさがって書かない側に寄ってサボる。

職場など複数人環境で PTTM をやる際の同僚からの「何してるの？」攻撃には、初期は適当にごまかし「そういうキャラ」として定着させる。

### 4: 管理対象を決める
PTTM はテキストを書くため何でも扱えるが、向いていないものもある。何を扱う **か**、より **何を扱わないか** を決める。

予定はカレンダー、日記や思考はだらだら書くと圧迫されて読み返さなくなり形骸化。「上手くいかない」場合はまず **何でも扱いすぎていないか** を疑う。詳細は [扱わないもの](#扱わないもの)。

### 5: 振り返りを行う
PTTM は読み書きとメンテナンスを行うものなので、[振り返り](furikaeri.md) の習慣が身についていると強い。

PTTM では: 残タスクの確認、明日以降のタスクの確認、完了反映、軽微な修正――を頻繁に行う。日 / 週 / 月単位で俯瞰した反省も。[DWMY](furikaeri.md#dwmy-と-dwmy-ヒエラルキー) の頻度を意識する。

部屋やデスクと同じく、テキストも散らかる。掃除しないと形骸化する。

### 6: 3 秒の壁を突破する

> 新しいタスクを追加したい場合、3 秒以内に追加することを目指す

3 秒以内にメモを **書き始められる** くらい素早くなれるなら、PTTM が手に馴染んでいる。

タスクを「後でやろう、覚えておけば大丈夫」と考えると忘れる。記憶能力の過大評価。その場でメモを完了させる必要がある。

> 「面倒くさい」が勝ってしまうとサボる

PTTM はちょっとしたサボりが肥大化して形骸化する。さっさとメモする場面はサボりやすい筆頭。3 秒の壁を壊すための工夫は惜しまない。

ナレッジワーカーやクリエイターなど創造的な人にとっては特に重要。アイデアはいつ湧くかわからず、湧いたときにメモしないと未来永劫忘れる。詳細は [メモ](memo.md)。

## 扱わないもの

### オルタスクは扱わない
[オルタスク](altask.md)（コンテナ / イベント / モットー / メモ / タスクソース）は PTTM では分が悪いので、各専用ツールで扱う。

ただし限定的には扱える:

- **[コンテナ](container.md)** — 見出し記法やインデントで階層的表現
  ```
  ■引っ越しエリア
  - 引っ越し
      - 引越し先の検討
          - 北海道を調べる
          - 地元を調べる
      - 全体スケジュール
          - 梱包や粗大ごみ
  ```
- **[メモ](memo.md)** — むしろ普段の PTTM ファイルに直接書くのが自然。ただし溜めずに、振り返りで処理する

### ルーチンタスクは扱わない
[ルーチンタスク](routine_task.md) を PTTM で管理するのは厳しい。

例: 毎週火曜と金曜のゴミ捨てを PTTM で管理する場合、毎日「今日が火金か」を確認して該当日にタスクを追加する作業が要る。やってられない。

ルーチンタスクは専用のツールで管理する（[Robot](robot.md) / [Tracker](tracker.md)）。

## 設計概念

### デイリータスクリストを導入する
タスクリストを日単位で区切る。今日が 2024/07/31 なら `■2024/07/31` で今日エリアを設けて、今日はそこに書いてあるタスクだけ終わらせれば OK。

> 人は終わりを設けてメリハリをつけないとガス欠する

詳細は [Dailer](dailer.md)。

### 曖昧な日付感
日単位で区切る際の 2 つのやり方:

| 方式 | メリット | デメリット |
| --- | --- | --- |
| **日付ベース** `■2024/07/31` | 見やすい、正確 | メンテが大変（日付エリア作成、今日を一番上に移動） |
| **意味ベース** `■今日 / ■明日以降` | メンテが軽い | 後回しが起きがち、タスクが溜まりやすい |

日付ベースで挫折するくらいなら意味ベースが良い。

意味ベース内で **時期的な広がり** を空行や区切り線で表すこともできる:

```
■明日以降
- ユーザーテスト
- バグ修正

- 仕様書更新
- チーム会議


- 新機能のアイデア出し
```

または:
```
■明日以降
- ユーザーテスト
- バグ修正
|
- 仕様書更新
- チーム会議
||
- 新機能のアイデア出し
```

### フォーマットを統一する
PTTM は文字情報ばかりで見づらい。挫折の主因の一つ。

> 少しでも見づらさを減らして負担を減らすべき

正解はないが、自分なりのレイアウト・記法を決めて守る。例:
- 日単位エリアは `■2024/07/31`
- 1 タスクは 1 行
- 各タスクは `- ` または `・` で開始

最初は苦しいが、目が慣れてくると素早く読めるようになる。**続けたいなら絶対にやっておきたい投資**。

ゼロから考える必要はなく、Markdown のような既存文法に頼っても良いし、[先人の実践例](#先人の実践例) からパクっても良い。模索の中で愛着が湧いてくるとモチベーションも支えられる。

### スペースは半角で統一する
半角スペース「` `」と全角スペース「`　`」では半角に統一すべき。

- 検索時にスペース種別が混在すると面倒
- エディタのインデント上げ下げ機能は半角前提（全角だと動かない）

全角分のスペースが欲しいなら半角スペース複数回。例外的に全角を入れたいときは Shift + スペースで切り替え（Google 日本語入力の場合）。常に半角にするには IME 設定が必要。
