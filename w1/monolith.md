# Monolith

タスクを一箇所に集める [戦略](strategies.md)。具体的には **一つのテキストファイルに** すべてのタスクを書く。一箇所にまとめる系。

Monolith = 一枚岩。IT 文脈では「何でも詰め込んだあり方」のたとえ。エンジニアやクリエイターなどテキストエディタによる書き込みに長けた人が、自分の文章で [Richild](richild.md) 風の発想をすると行き着く形。

## 例: stakiran.txt

```
■今週のタスク
...

■引っ越し検討
...


■ ---


■2024/05/24
...

■2024/05/23
...
```

`■` で塊を区切る。上が作業領域、`■ ---` 以下が過去ログ。Monolith はこのファイルを生活動線として日々過ごす。終わったタスクは下に移すか別ファイル（stakiran_old.txt）に退避。

文章入力スピード、言語化スキル、テキストエディタ操作・カスタマイズの知識を備えるので手足のように扱える。テキストエディタは強力で数百万文字まで耐える。バックアップや同期も 1 ファイルだけ。

著者は秀丸エディタの愛用者で、一時期 Markdown ベースの Monolith を使用。見出し記法でアウトライン化していた。

## 応用

### Task Monolith vs Note Monolith
- **Task Monolith** — タスクのみ。現代は SaaS 充実で多数派
- **Note Monolith** — タスク以外も全部（日記、ネット見て気になったこと、問い合わせ下書きなど）

詳細は [PTTM（プレーンテキストタスク管理）](pttm.md) に繋がる。

### クラウドツールでの Monolith は可能か
**難しい**:

- Monolith は手足に馴染んだ操作性が前提（0.1 秒の違和感もない）
- SaaS にはオンライン遅延、テキストエディタほどのカスタマイズ性は無い
- 数十万文字レベルのテキストをスムーズに扱える安定感もない
- 今後も至らない（生成 AI 時代は書くより見る・微修正が中心になる）

### 予定や日課・習慣の扱い
テキストだけでは扱いづらい:

- 予定 → カレンダー
- 日課・習慣 → Todoist、タスクシュート、習慣トラッカー

どうしても Monolith でやりたいなら自作カスタマイズ。**かなり分が悪い**。三次元立体を二次元平面に変換するような感覚。

## メリット
- 1 テキストファイルで済むシンプルさ
- テキストエディタのベテランにフィットする

## デメリット
- 軽快な操作・タイピング・言語化が要求され、合う人が少ない
- 時代の流れに沿っていない、今後の発展なし
- 予定・日課・習慣など、テキストでは扱いづらい性質のタスクが扱いづらい

## 適性
- **日頃からテキストエディタを愛用している人**。普段の文字入力でテキストエディタが最多
- **エンジニア・プログラマーは意外と向いていない** — テキスト自体への傾注が必要
- ブロガー、作家の方が向いている可能性
