# タスクの属性

タスクが持つ属性のリファレンス。タスク管理ツールやメソッドはこれらの属性をどう選び、どう見せるかが設計の中核となる。

## タスクの 3A

タスクは 3 つの要素から構成される:

| 要素 | 英語 | 内容 |
| --- | --- | --- |
| 属性 | **A**ttribute | タスクの構成要素（名前、状態、締切など） |
| 関係 | **A**rrow | 他タスクとの関係（包含、参照、依存） |
| 機能 | **A**ttachment | タスクが持つ各種機能（コメント、添付、リマインダー等） |

> [プロジェクトタスク管理の 3A](3a.md)（Assign/Adjust/Alternate）とは別物。同名異概念なので注意

### 関係の 3P
**3P** で整理:

- **Parent-Child（親子関係）** — 親タスクに子タスクがぶら下がる。粒度の分解にも使う。最も身近で網羅性が高いが負担も大きい
- **Point（ポイント関係）** — リンクでつなぐ。タスク管理では稀（[Scrapbox](topician.md) や Obsidian など）。網羅性は劣るがゆるく続けやすい。**文芸的タスク管理**（後章 literate.md）で詳述予定
- **Pre and Post（依存関係）** — タスク A は B と C 完了後に開始可能、というワークフロー的な依存。設計が必要だが上手くいくと最も楽。**IFTTT / Zapier** やローコード/ノーコードと同じ発想

### 機能（Attachment）の例
コメント / 添付ファイル / チェックリスト / リマインダー / リアクション。

> 機能の利便性や世界観に惑わされて、タスク管理をするのだとの本質を見失わないようにしてください

タスク管理ではまず **関係と属性** に目を向ける。機能は補助。ただし [パートナータスク管理](partner_task_management.md) の Chat per Task のように、機能が主役級になるケースもある。

## 属性のジレンマ

> 属性が多いほど、できることも増える代わりにメンテナンスコストも増える

トレードオフ。重要なのは **自分にとって必要な属性だけをコンパクトに取り入れる** バランス。

## 属性の 9 系統

| 系統 | 含まれる属性 |
| --- | --- |
| 記述系 | 名前 / 詳細 |
| 状態系 | 状態 |
| 識別系 | ID / 番号 |
| 日付系 | 実行日 / 作成日 / 締切日 / 完了日 |
| 時刻系 | セクション / 開始時間 / 終了時間 / 見積時間 / 実績時間 |
| 定期系 | 繰り返し頻度 / 繰り返し条件 |
| 分類系 | カテゴリー / タグ / ラベル / スター / 優先度 / トリガーコンテキスト |
| 主観系 | 重要度 / 難易度 |
| 協調系 | 担当者 / 発行者 / 貢献者 / 購読者 |

## 記述系属性

タスクの内容を表現する。表現には言語を用いる。

### 名前 (Name)
タスクの名前。1 行・100 文字以内程度のテキスト。**文章** にして具体的にすると扱いやすく取り組みやすい。

todo.txt のように名前の中に他属性を持たせる流派もある（`(A) 2017-07-14 先方にお礼の電話をする @phone +HogeProj`）。**ツールが処理（ソート/フィルタ）できるなら属性、人が読むだけなら名前**。

### 詳細 (Description)
複数行のテキスト。プロジェクトタスク管理など本格的なツールで使われる。個人レベルでは持て余しがち。

ベストプラクティス: なるべく使わない / 分量を少なくする / テンプレート化して統一する。

## 状態系属性

### 状態 (Status)
未着手 / 進行中 / 完了 など。

理論的には「開始前」「開始後」「終了前」「終了後」の 4 ステップがある。ペンディングは「終了前」に属する。

最低限 **未完了/完了の 2 値** は欲しい。完了タスクは削除 or 退避（ログ化、アーカイブ化）するべき。混ざっていると見るたびに認知コストを消耗する。

更新は頻繁な操作なので **クリック 1 回 / キー入力数回以内** で行えるよう設計する。

## 識別系属性

### ID
プログラム内部で識別するもの。利用者は通常見ない。同名タスクを複数扱える（個人ではほぼ用途なし、タスクごとの専用ページがあるツールで意味を持つ）。

### 番号 (Number)
ID の一種だが **連続英数字 + 順序性** がある。番号が若いほど先に実行する。例: タスクシュート（Excel 版）は連番ソートで並び順を維持する。

## 日付系属性

日付の表記は様々（240722, 2024/07/22, 2024-07-22 など）。曜日は計算で出せるので通常含めないが、表示しないと読みづらい場合は工夫を。

### 実行日 (Starting Date)
タスクを実行する日。**計画と先送り** に使える:

- 計画: 大きな何かを分解した小タスクを日ごとに配置
- 先送り: [Dailer](dailer.md) で「今日のタスクを別の日に動かす」

カレンダーは実行日属性付きタスクの格子状表示。**カレンダーを超える俯瞰方法を人類はまだ編み出していない**。

### 作成日 (Creation Date)
通常は不要。自動化前提だと「n 日以上未着手なら自動 close」のような Bot 処理に使える（GitHub Issues など）。

### 締切日 (Due Date)
個人タスク管理では形骸化しがち。**実行日属性で代替できる**（大きいなら分解する）。

プロジェクトタスク管理（綿密な計画）では重宝。アジャイルでは無用に近く、「実行週」属性なるものが欲しいところ（事例なし）。

### 完了日 (Completion Date)
[振り返り](furikaeri.md) に役立つ。実行日や締切日との比較で実績や守れた率がわかる。状態属性の代替にもなる（記入されたら完了とみなす、todo.txt が好例）。

## 時刻系属性

日単位より細かい単位で管理する。日単位で済むなら不要。

### セクション (Section)
時間帯。実行日と開始時間の中間粒度。詳細は [Slotter](slotter.md) と [コンテナ](container.md) を参照。

値の例: 出社前/出社中/出社後午前/昼休憩/午後 1/午後 2/退社中/退社後/就寝前。**自分にとって最適な値** を定義する。

### 開始時間 (Starting Time)
終了時間と組み合わせて実績時間を計算する。記録は **ボタン 1 回程度** に省力化が必須。

並列実行の扱いを事前に決めるべき（通常は不可、計算がおかしくなる）。単位は内部秒、表示分が望ましい（切り替えできるとなお良い）。

### 終了時間 (End Time)
連続実行なら次のタスクの開始で省力化可能。終了時に追加アクション（感想記入、繰り返しなど）を行うかどうかを設計。

**キャンセルタスク** の扱いも検討（開始時間消去か、終了で打ち切りか）。

### 見積時間 (Estimation Time)
タスクごとの見込み時間。デイリータスクに対して **All or Nothing** で運用する（一部だけだと意味がない）。

### 実績時間 (Actual Time)
開始と終了の差分。記録の正確性に依存。

## 定期系属性

[ルーチンタスク](routine_task.md) を扱うための属性。

### ルーチンタスクの設置方式 3 つ
| 方式 | 内容 |
| --- | --- |
| **Set all firstly** | 設定値に基づいて該当する未来日すべてに設置（カレンダーで使う） |
| **End and Set** | 終了時に次のタスクを生成（タスクシュート） |
| **Counter** | とりあえず毎日表示、利用者がやる/やらないを決める（習慣トラッカー、Habitica） |

### 繰り返し頻度 (Repeat Frequency)
「n 日ごとに x 回」。x=1 で運用するのが扱いやすい（複数回なら 1 日 1 回タスクを n 個つくる）。日以上の単位で扱う。

### 繰り返し条件 (Repeat Condition)
「毎週水・金」「毎月 30 日」「第 2 水曜日」など、頻度では表現できない条件。

開発負担が大きい（複雑な日付計算、文化固有の制約）。**Manual Listup**（出現させたい日を利用者が選ぶ）方式も実用的。

## 分類系属性

詳細は [コンテナ](container.md) を参照。

> 分類項目がタスクを包含（メタ包含）するのに対して、3A における関係の親子関係はタスクがタスクを包含（同族包含）する

### カテゴリー (Category)
タスクに 1 つだけ付与可能。**事前に注意深く設計して厳格に分類** するスタイル。乱用しない。

### タグ (Tag)
複数付与可能。SNS のハッシュタグと同じ感覚。**タスク管理ではたくさんつけない**（フィルタリングが目的なので、多すぎると探せない）。

### ラベル (Label)
**色付きタグ**。視覚的な識別の早さに特化。GitHub Issues、トリアージ（黒赤黄緑）。

### スター (Star)
1 つだけ付与可能、CRUD なし（つけるかつけないかの二値）。「あとで読む」「未読」など [インボックス](inboxer.md) 用途。

形骸化しやすい属性。**外すコストの高さ** が原因なので、メーラーのように **自動で外す** ように倒すのが望ましい。

### 優先度 (Priority)
1 つだけ付与可能。値は高/中/低、5/4/3/2/1、Emergency/Alert/Critical/...など。

個人タスク管理では通常不要。プロジェクトタスク管理向き（定義としっかりした運用が要る）。個人で感覚的に区別したいなら **スター** で十分。

### トリガーコンテキスト (Trigger Context)
詳細は [コンテキスト](context.md) と [コンテキストタグ](context_tag.md)。

通常はタグ/ラベルとして実現。**複数付与は推奨せず 1 つに絞る**（複数あると判断機会が増えて疲れる、形骸化する）。

## 主観系属性

定量的に決めるのが鉄板（文章は記述系か機能で扱う）。

### 重要度 (Importance)
利用者にとって重要なことを可視化。本質的には主観で運用する優先度。

> 重要の定義に重要以外の意味（緊急性、収益性、稀少性）を含めない

[時間管理マトリクス](shallow_deep.md) では重要と緊急を区別している。

### 難易度 (Difficulty)
取り組み方の調整に使う。定義はさまざま: 不確定性 / 技術的難易度 / 時間的難易度 / 好き嫌い / [ロードコンテキスト](load_context.md) の量 / 拘束の程度。

**事後評価には使わない**（事前評価専用）。

## 協調系属性

[プロジェクトタスク管理](project_task_management.md) で使う。個人タスク管理では基本不要。

### 担当者 (Assignee)
タスクの実行責任者。「担当者=自分」フィルタで自分の抱えるタスクを抽出できる。

### 発行者 (Owner)
タスクを作成した人。自動記録するべき。**通常使われない**（使われている場合は管理が複雑化、または政治発生のサイン）。

### 貢献者 (Contributor)
タスクに携わった人。BTS で議論が活発なときに「誰がどの議論に絡んでいるか」を確認するのに役立つ。

**貢献者属性のジレンマ**: 視認性は上がるが、特定の人のタスクだけ読む / 嫌いな人のタスクは無視する、といった政治色を強める危険性も。

### 購読者 (Subscriber, Watcher)
タスクの更新通知を受け取る。RSS や SNS の発想。

> 購読者属性を重視したタスク管理は現在開拓されていない

著者の見立て: BTS のような議論型・ティール組織と相性が良いはずで、**キラーアプリのポテンシャル** がある未開拓ジャンル。

## ツールのパラドックス

属性が代替可能でも、**しっかりサポートしたツールを別途増やしてでも使う** 方が、力不足なツールで代替するより定着しやすい。

> 心理的には今のツールで使いたいと考えがちだが、力不足な代替では形骸化しやすく定着しない

キッチン用品も一種類より複数の使い分けの方が上手くいく。直感には反するが正解。
