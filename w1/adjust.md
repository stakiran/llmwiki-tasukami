# Adjust — 調整

タスクの属性値を変えること。プロジェクトタスク管理 [3A](3a.md) の 1 要素。

## 属性とは

タスクが持つ特徴。例:

- 名前
- 状態（未着手 / 着手中 / 完了 / 中止）
- 担当者
- 締切 / 開始日
- 優先度（高/中/低、1-5 など）
- 費やした時間（工数）
- コメント

> プロジェクトタスク管理とはタスクの属性値を変える営みともいえる

タスクは勝手に変わらない。能動的に変えるしかない。記録には [ツール](axioms.md) が要る。

## 調整は頻発する

イレギュラーがつきものだし、慌ただしいと調整量は増える。新タスク、消えるタスクは平気で起きる。これらを全部ツールに反映できないと、誰かの脳内に留まる → 引き出すための会話 = 会議が必要になり、人によって認識が違うので齟齬が起き、時間と労力が消える。

> プロジェクトは大変になりがちだが、その主因は単にツールに反映ができていないことによる秩序の崩壊

## 反応的状況（Reactive Situation）

その時その時で起きたことに即座に反応する形で生き延びていく状況。慌ただしい世界は基本的にこれ。

- 調整（とツール反映）は追いつかない
- 目の前を見て瞬発的に反応するしかない
- → タスク管理が通用しない
- 逆に言えば、タスク管理という面倒くさいことをしなくて済む
- だから反応的状況はよく好まれる。プロジェクトが無駄に忙しくなりがちな一因

調整の頻発に対抗する手段は、現実的には 2 つしかない:

1. ツールへの反映を徹底する（通常そこまでのスキルは無く非現実的）
2. 反応的状況に溺れる
