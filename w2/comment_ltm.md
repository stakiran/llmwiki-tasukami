# LTM へのコメント

対象: [w1/LTM（文芸的タスク管理）](../w1/ltm.md)

## ライティングスタックは個人 SE の生命線

w1 はこう書く。

> LTM が定着するかどうかは、ライティングスタックの考え方に染まれるかどうかにかかっている
> — [w1/ltm.md](../w1/ltm.md) 「ライティングスタック」

ゆるい知的生産の立場からも、これには大きく頷ける。**頭の中ではなくノートを原本にする** という規律は、ストイックさを要求する規律ではなく「忘れても大丈夫にする仕掛け」であり、まさに [raw2/ゆるい知的生産.md](../raw2/ゆるい知的生産.md) で言う「泥酔してても使えてほしい」の実装である。

頭を信用しない、というのは性格の問題ではなく **設計上の選択**。これを「がんばって書く」のではなく「書かないと自分が困るから自然に書く」運用に持っていけたら、LTM はゆるい知的生産そのものになる。

## 切り出しが文脈をつくる

> 切り出しが文脈をつくる
> — [w1/ltm.md](../w1/ltm.md)

ここが LTM の最重要パートだと思う。文章中の語句をそのままノート名にしてリンクを引く、という操作は、Scrapbox / Obsidian / Cosense の文化そのもの。raw2 で言う「井戸端」状態は、まさにこのリンクの引き合いから立ち上がる。

個人 SE の場合、これはコードベースの design doc / RFC / ADR を切り出していく営みと地続き。**切り出した瞬間に判断や決定が部品化される**ので、後で別の判断と接続できる。階層整理を頑張らなくても、リンクで自ずとネットワークができる。

> 文芸的タスク管理は、一応タスク管理といいつつも、本質はノートテイキング
> — [w1/ltm.md](../w1/ltm.md) 「文芸的タスク管理の本質」

これも完全に同意。タスク管理を冠しているけれど、ゆるい知的生産から見れば、これは **「ノートテイキングをきちんとやれば自ずとタスクは見える」** という主張に近い。

## LLM 時代の LTM

ここから先は w1 には書かれていない、自分の問い。

LLM が出てきたことで、「第二の脳」の場所が変わりつつある。LTM の世界観は「ノート＝原本」「頭は作業場」だったが、LLM 時代には:

- LLM がノートを読んで文脈を組み直してくれる（要約・蒸留が自動化される）
- 「どのノートを今読むべきか」を LLM が選んでくれる
- 切り出し候補も LLM が提案してくれる

つまり LTM の **「ノートを読み返す労力」** が大幅に下がった。これはゆるい知的生産にとっては追い風で、「読み返しの土地勘がついていなくても、LLM が代行してくれる」状態になる。

ただし注意点として、LLM に任せ切ると **自分で書く動機が落ちる** 危険がある。LTM の本質は書くことで文脈をつくることなので、書かなくなったら LTM は崩壊する。LLM は読み返しと再構成のレイヤーで使い、書く部分は自分でやり続ける、というのが現時点での暫定解だと考えている。

## アンカバードを認める潔さ

> アンカバードのやり方ばかりに頼っていては LTM はできない
> — [w1/ltm.md](../w1/ltm.md)

ルーチンタスクや締切厳守は LTM ではカバーしない、と明言しているのは好感が持てる。「全部 LTM でやる」と言わない潔さ。ゆるい知的生産も同じで、**「ゆるさで全部押し通す」は思想であって運用ではない**。締切は別ツール、リマインドは別ツール、ゆるさを担保したいパートだけ LTM、という分業が現実的。
