# ハビットエンジニアリングへのコメント

対象: [w1/ハビットエンジニアリング](../w1/habit_engineering.md)

## 振り返りと並ぶ「ストイック疑念」の対象

ハビットエンジニアリングは、たすかみの中で **振り返りと並んで「教える人が必要な手法」化しやすい** 章だと感じる。GSAAD（Get / Set / Act / Adjust）、ハビットスタッキング、IF-THEN プランニング、行動パラメーター、着手ベース、ハビチュエーション、撤収、捨て身……語彙が一気に増える。

[raw2/ゆるい知的生産.md](../raw2/ゆるい知的生産.md) のフレーズを当てると:

- 「マナー講師がマナーを作る」可能性が高い領域
- 「頑張ってる自分を自慢したい人」が好みやすいフレームワーク
- 「実際にやってる人」は、たぶん GSAAD を意識せずに勝手に習慣化している

つまりゆるい知的生産の立場からは、**この章は批判込みで読む** のが正しい接し方だと考えている。

## ただし「内容」は否定しない

語彙の厚さに引いてしまうけれど、書かれている内容自体は妥当で、しかも **ゆるい知的生産的な思想に支えられている** 部分が多い。

w1 から:

> 解は「モチベが湧くまで諦めなさい」あるいは「別の、モチベが湧く行動をターゲットにしなさい」
> — [w1/habit_engineering.md](../w1/habit_engineering.md) 「Adjust」

> 重要なのは行動の質ではなく行動の回数
> — [w1/habit_engineering.md](../w1/habit_engineering.md) 「着手ベース」

> 自分が実際に行動に起こせる程度の強度を設定すること
> — [w1/habit_engineering.md](../w1/habit_engineering.md) 「Set」

これらは全部、根性論や精神論の真逆。**「無理なら諦めろ」「強度を下げろ」「回数で勝負しろ」** というメッセージは、ゆるい知的生産の現場感覚と一致する。

ストイック側の本なら「歯を食いしばって続けろ」と書くところを、たすかみは「続かないなら別のターゲットにしろ」と書いている。これは raw2 で言う「実際にやってる人がやってること」を書く側の姿勢。

## GSAAD は「振り返ったときの語彙」として使う

ゆるい知的生産的な接し方の提案: **GSAAD は事前計画ではなく事後の言語化に使う**。

- 何かが習慣化した → あとから「これ Get-Set-Act-Adjust 通ってたな」と振り返る
- 何かが続かなかった → あとから「Set が弱かったな」「行動パラメーターがきつかったな」と分解する

GSAAD を **最初から仕掛けようとすると重い**。だが、振り返りの語彙として使うと、自分の試行錯誤を整理する補助になる。「教える人が必要な手法」から「実際にやってる人が後付けで使う語彙」に降ろす運用。

これは ETM の「ゴールはあとから見えてくる」の構造と同型で、ゆるい知的生産の典型パターン。

## 「捨て身」の正直さ

w1 が誠実だと感じるのは、最後のここ。

> 解はあまりない:
>
> - Set の項で挙げた 環境や人など外部の力に頼った仕掛け
> - 撹乱（環境ごと変える）
>
> 職人の弟子入り、スポーツのクラブ活動、寮生活、合宿などでは生活を厳しくコントロールされ年単位で続くので、嫌でも習慣が身につく
> これらは 荒療治。本来、習慣はそれくらいの荒療治が必要。地道なハビットエンジニアリングができないなら、荒療治しかない = 捨て身。
> — [w1/habit_engineering.md](../w1/habit_engineering.md) 「捨て身」

ここがゆるい知的生産から見て一番面白い。**「そもそもハビットエンジニアリングは万人向けではなく、できないなら環境ごと変えろ」と言い切っている**。

これは raw2 の「実際にやってる人」「教える人」の二項対立に、第三項として「**やれない人**」を加えた話。やれない人にフレームワークを売る、というのがマナー講師ビジネスのパターンだが、たすかみは「やれない人は荒療治しかない」と正直に書いている。

ゆるい知的生産の立場としては、**自分が「やれない人」であることを早く認める** のが利口。捨て身（環境を変える）の方が、続かないハビットエンジニアリングを延々と試みるよりずっと結果が出る。

## 「人間が変わる方法は 3 つしかない」

w1 が引用している有名な命題:

> 人間が変わる方法は 3 つしかない: 時間配分、住む場所、付き合う人
> — [w1/habit_engineering.md](../w1/habit_engineering.md) 「3. 撹乱」

これはゆるい知的生産の最終手段の言語化。井戸端的な世界に身を移す、引っ越す、付き合う人を変える — これで生活が変わる。ハビットエンジニアリングのスタックを積むより、environment を切り替える方が桁違いに効く、という主張。

独立した個人 SE には、組織人より明確に効く。住む場所と時間配分と付き合う人をすべて自分で選べる立場なので、習慣化に詰まったら **環境を変えるのが一番安い** ことが多い。

## オートマトンとの違い

[w1/オートマトン](../w1/automaton.md) も「仕組みづくり」の章だが、ハビットエンジニアリングとは温度が違う。オートマトンは「楽したい」が動機、ハビットエンジニアリングは「変わりたい」が動機。

ゆるい知的生産的にどちらが居心地良いかと言えば、**オートマトン側**。ハビットエンジニアリングは「変わる必要があるという前提」が入っており、そこに「ストイックを志向する自分」がうっすら混入する余地がある。

「変わらなくていいなら変わらない、変わるなら環境ごと変える、中途半端な習慣化はやらない」がゆるい知的生産的な構え。GSAAD の精緻なループは、それでも変えたい一部の習慣のために棚から取り出す道具、くらいの位置付けでよい。
