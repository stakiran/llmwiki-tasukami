# 気付き: モードを分けるからゆるさが守れる

ETM のコメントとデリゲーター系のコメントで矛盾している気がした論点を、整理して再構築したもの。

## 主張

ゆるい知的生産は「常にゆるく」運用するのではない。**生活や仕事をいくつかのモードに分け、ゆるくするモードと、瞬発力で片付けるモードを切り替える** ことで、ゆるさが守れる。

混ぜようとすると:

- ゆるいモードの時間にストイック作業が侵食する
- ストイックモードの時間にゆるさが入って雑になる
- 結果として **どちらも中途半端** になる

これは [w1/ETM](../w1/etm.md) の「探索モードと指向モードを同時に行わない」と同じ構造。「同時に走らせる」を批判していた自分の最初の見解は、**部分的に間違っていた** ので訂正する。

## 出どころ

- [comment_etm.md](comment_etm.md): 当初 ETM の「同時に走らせない」規律を「ストイック側に倒れている」と批判した
- [comment_delegator_sprinter_skipper.md](comment_delegator_sprinter_skipper.md): 一日のうちアスリートモードで雑務を 3 時間で片付け、残りはゆるく井戸端的に過ごす、という構図を肯定した
- [w1/etm.md](../w1/etm.md):
  > 重要なのは両者を同時に行わないこと

最初は批判したが、後で別文脈で同じ構造を肯定していた。**自分の中で矛盾していた**。

## 矛盾の解消

最初の批判は「同時に走らせる方が井戸端的で自然」というものだった。しかし井戸端と仕事を完全に混ぜると、現実には:

- 仕事中の Slack 雑談で本来の集中が崩れる
- 雑談中に仕事の依頼が降ってきて雑談が消える
- 結果、ゆるさも仕事の質も両方落ちる

ゆるい知的生産が成立するためには、**「ゆるくいられる時間と場所」が物理的に確保されている必要がある**。これは raw2 の masuilab や井戸端の場が、**仕事チャネルとは別のプロジェクト** として独立して存在していることと同じ。

つまり最初の批判は、**「同時に走らせる」を「混ぜる」と取り違えていた**。

正しい整理:

- **混ぜる** ❌: ストイックとゆるさが時間と場所で混在し、互いを侵食する
- **同時に走らせる** ⭕️: 別々の時間・別々の場所・別々のチャネルで並行する
- **片方ずつ** ⭕️: 時間帯やモードで切り替える

ETM の「同時に行わない」は **「混ぜない」** を意味していたと読み直せる。

## モードの実例

ゆるい知的生産を支持する個人 SE が運用するモード分けの例:

| モード | 時間帯 / 場所 | 内容 |
| --- | --- | --- |
| **アスリートモード** | 午前 / オフィス時間 | 即返信、即着手、雑務一掃、コードレビュー |
| **指向モード** | 午後 / 集中時間 | 設計、実装、合意した仕事 |
| **ゆるモード（井戸端）** | 夜 / 週末 / 散歩中 | Scrapbox 書き散らし、Discord 雑談、OSS、フリーライティング |
| **完全休息** | 不定期 | 何もしない、寝る、犬と過ごす、本を読む |

これらは **混ぜない**。アスリートモードの時間に Scrapbox を眺めない、ゆるモードの時間に顧客 Slack を見ない、など物理的な分離が要る。

## 含意

- ゆるい知的生産は「常時ゆるい」ではなく「**ゆるい時間が確保されている**」状態
- モード切替のコストはゼロではないが、**混在のコストの方がはるかに高い**
- 通知制御、デバイス分離、時間ブロック、場所の使い分けが運用基盤になる
- これは [リジェクター](../w1/rejecter.md) の拒否ルール（「17 時以降の通知は無視」など）と同じ構造

## たすかみとの整合

たすかみの中で、モード分けを支える概念は多い:

- [w1/Inboxer](../w1/inboxer.md) — 動線をひとつに絞ることで他モードを侵食させない
- [w1/Slackist](../w1/slackist.md) — 余裕を確保し、ストイック時間で侵食させない
- [w1/Rejecter](../w1/rejecter.md) — 拒否ルールでモード越境を防ぐ
- [w1/Automaton](../w1/automaton.md) — モード切替を自動化する

たすかみは全体として **「混ぜない」運用** を様々な角度から書いている本だと読める。最初は「カテゴリが多すぎる」と感じた厚さも、実は **モード分離のための語彙集** として機能している。

## 注意点

- モード分けは硬直化すると逆効果（緊急時に切り替えられない、リズムが固定されすぎる）
- モードを多くしすぎると切り替えコストが嵩む。3 〜 4 モードが目安
- 個人差が大きい。瞬発力タイプ / 持久力タイプで適正なモード構成は変わる

つまりこの気付きは **「モード分けはゆるさを守る装置だが、装置自体がストイックになると本末転倒」** という二重否定の構造。モード分けも事後の語彙として軽く扱うのが正解 → [derived_framework_post_hoc.md](derived_framework_post_hoc.md) に接続。

## 派生する問い

- モードの最低数 / 最大数はどのくらいが妥当か
- LLM がモードを跨いで存在することの是非（ゆるい雑談 LLM とアスリート LLM を分けるべきか）
- モード切替の「儀式」（コーヒー、散歩、服を変えるなど）はどこまで効くか
